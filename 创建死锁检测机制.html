创建死锁检测机制

死锁检测器 (Lock Dependency Validator，简称 lockdep)是内核最强大的调试特性之，主要用于调试内核同步问题。死锁检测器会记录系统中所有锁的申请顺序，构建依赖图。如果发现“A等B，B又等A”的潜在风险，它会立即报警。内核在初始化过程中会非常早地使用各种锁（如 slab_mutex 或各硬件驱动的自旋锁），如果不把死锁检测机制建立起来，早期代码中的死锁风险就无法被监测到。

由于死锁检测机制需要维护复杂的依赖图，会消耗额外的 CPU 和内存，发布版的内核通常会关闭 CONFIG_LOCKDEP。死锁检测是逻辑检测，只要代码逻辑中存在“产生死锁的可能性”（违背了加锁顺序），它就会直接在大黑屏上打印出长长的堆栈信息（Backtrace）。

lockdep_init() 是 Linux 内核死锁检测器的初始化函数。 在内核启动早期调用它，是为了给后续所有的锁操作建立“监控网”。其主要任务是初始化内核用来追踪锁依赖关系的哈希表和链表，建立依赖图（Dependency Graph），准备基本全局变量等。哈希表主要有用于统计锁的持有次数、竞争情况的lock_stats，用于按锁的类型追踪的class_hashes，用于追踪锁的持有顺序的chain_hashes。

函数位于文件git/kernel/locking/lockdep.c中，由一系统printk()打印函数组成，这里就不列出该函数的定义，有兴趣的读者可以打开文件浏览。

从函数定义可以看出，该函数只是通过printk()打印一些死锁检测器的有关信息，并没有进行初始化操作。这是由于这些都是静态定义的链表，它们在编译期就已经分配好了内存，而且很多链表在定义时已经初始化，或者在静态数组中默认初始化为 0，所以在lockdep_init()实现中，不需要再初始化这些结构。

在建立死锁检测机制后，函数locking_selftest()通过“故意制造错误”来测试 Lockdep（死锁检测器）是否真的能干活。它会模拟各种非法的加锁场景，验证 Lockdep 能否准确识别并报警。它测试的场景包括：

  • A-B / B-A 死锁，即先拿 A 再拿 B，另一个线程先拿 B 再拿 A。

  • 循环依赖，即A -> B -> C -> A。

  • 递归加锁，即同一个线程对同一个非递归锁加锁两次。

  • 中断上下文违规，即在硬中断里拿了只有进程上下文才能拿的锁。

locking_selftest()函数位于文件git/lib/locking_selftest.c中。下面简单介绍函数的主要功能。

在确定系统允许测试（利用debug_locks遍历）后，设置允许读递归锁定变量force_read_lock_recursive，打印测试表头，然后利用函数init_shared_classes()初始化锁类别，并通过函数lockdep_set_selftest_task()把测试线程设为当前线程。

接下来的工作是利用宏DO_TESTCASE_XX与函数dotest()等进行各种锁测试。宏通过函数或其它宏定义而成，同样定义在locking_selftest.c，调用格式为DO_TESTCASE_XX (desc,name)。desc为描述测试组合的字符串，name为测试函数模板，name_type构成指向实际测试函数的指针，如DO_TESTCASE_6R(“A-A deadlock”, AA)将会调用AA_spin()、AA_wlock、AA_rtmutex等测试函数。测试函数由宏GENERATE_TESTCASE() 自动生成，如GENERATE_TESTCASE(AA_spin)生成函数AA_spin。宏GENERATE_TESTCASE的定义为：

#define GENERATE_TESTCASE(name) static void name(void) { E(); }

其中，E为每次调用GENERATE_TESTCASE()前定义的一个宏，用以定义实际运行的测试序列，如在调用GENERATE_TESTCASE(AA_spin)之前，会利用如下宏定义：

#define E() \

LOCK(X1); \

LOCK(X2);

对于spin_lock，宏LOCK定义在文件git/bin/lock-selftest-spin.h，其定义为：

 #define LOCK L

而宏L位于文件git/lib/locking-selftest.c，定义为：

 #define L(x) raw_spin_lock(&lock_##x)

E()最终展开为：

 raw_spin_lock(&lock_X1);
 raw_spin_lock(&lock_X2);

这样，GENERATE_TESTCASE(AA_spin)最终会生成一个如下的测试函数：

void AA_spin(void)

{

 raw_spin_lock(&lock_X1);

 raw_spin_lock(&lock_X2);

}

其它类型的测试函数也是以相同的方式生成。

DO_TESTCASE_XX 其实是一个 编码式命名。以DO_TESTCASE_6x1RRB为例，各部分含义如下：

部分 含义

DO_TESTCASE 执行一个测试用例

6 测试 6 种锁类型

x1 每种锁运行 1 次

RR read-read 关系

B 复杂锁依赖场景（通常含分支或循环）

简单理解就是执行一个针对读锁依赖关系的测试，在6 种类型上各运行一次。在文件git/lib/locking_selftest.c中定义了6 类锁，分别为：

编号        锁类型

1           spinlock

2           write lock

3           read lock

4           mutex

5           write semaphore

6           read semaphore

宏会自动在这 6 种锁类型上运行测试。

在执行完中断上下文死锁测试后，locking_selftest()调用函数ww_tests()测试 抢占/等待（Wound/Wait）互斥锁机制是否正确工作。它主要验证 Wound/Wait Mutex 的死锁避免算法。

在自测结束后，清除当前死锁检测器自测任务标记，并恢复正常打印死锁检测错误信息。恢复正常锁调试环境后，死锁检测器再检测到锁问题就会正常报警。
