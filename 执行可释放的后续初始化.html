基于ARM的Linux操作系统引导详解

执行可释放的后续初始化

kernel_init()函数首先调用kernel_init_freeable()执行后续仍需完成的内核初始化工作。此时PID 1 仍处于内核态，继续进行初始化工作。它是1号进程PID 1 在变成systemd 之前所作的最后的初始化工作，是 start_kernel() 后半场最后一块拼图。函数位于git/init/main.c，定义为：

static noinline void __init kernel_init_freeable(void)
{
	wait_for_completion(&kthreadd_done);
	gfp_allowed_mask = __GFP_BITS_MASK;
	set_mems_allowed(node_states[N_MEMORY]);
	cad_pid = task_pid(current);
	smp_prepare_cpus(setup_max_cpus);
	workqueue_init();
	init_mm_internals();
	rcu_init_tasks_generic();
	do_pre_smp_initcalls();
	lockup_detector_init();
	smp_init();
	sched_init_smp();
	padata_init();
	page_alloc_init_late();
	page_ext_init();
	do_basic_setup();
	kunit_run_all_tests();
	console_on_rootfs();
	if (init_eaccess(ramdisk_execute_command) != 0) {
		ramdisk_execute_command = NULL;
		prepare_namespace();
	}
	integrity_load_keys();
}

该函数完成剩余内核初始化、设置系统运行环境，为用户空间 init 进程做准备。由于该函数完成的工作比较多，我们将分两小节进行介绍。

函数首先通过函数wait_for_completion()等待2号进程准备就绪。因为kernel_init_freeable()会开始大量创建其他内核线程，而这些线程都要靠 2 号进程来统一管理和派生，所以它必须等待2号进程准备就绪才能往下继续运行。当2号进程准备就绪后，reset_init()会通过函数complete(&kthreadd_done)标志2号进程已准备就绪。

gfp_allowed_mask 是系统允许使用哪些 GFP 分配标志的全局总开关。它等于__GFP_BITS_MASK意味着现在系统可以使用所有合法的GFP 分配标志。

函数set_mems_allowed()的作用是修改的是当前 task 结构体的 mems_allowed字段（允许内存节点掩码），允许PID 1 在任何内存节点分配内存。

函数接着初始化各类核心子系统，包括：

  • workqueue子系统初始化，主要完成：

  • NUMA 亲和性绑定 ，确保各运行池（worker pool）知道所属于NUMA 节点，以便就近访问内存

  • 如果某个队列设置了 WQ_MEM_RECLAIM 标志，创建一个救援者线程。当内存极度紧张、无法创建新线程时，就会允许救援者线程，确保关键任务不会卡死

  • 为每个池子创建第一个内核线程（kworker）。从此以后，那些排队等待的任务（Work Items）才真正开始被执行

  • 开启看门狗，如果某个工作项霸占 CPU 时间太长，看门狗会发出警告。

  • 内存管理内部结构初始化，主要完成：

  • 创建内存专属队列，确保内存回收任务在任何时候都能跑

  • 注册热插拔CPU 状态回调函数，以便初始化或清理该 CPU 的内存统计数据

  • 更新每个内存节点上在线 CPU 的状态

  • 启动牧羊人定时器，以便在CPU 上的内存统计数据积压时强制将其同步到全局计数器

  • 导出调试接口 (/proc 文件系统)，以便用户通过命令像cat /proc/buddyinfo 看到数据

  • task RCU机制（一种按 task 执行流来等待宽限期的 RCU 模型）初始化，主要完成：

  • 创建用于 ftrace 等跳板指令回收的普通task RCU 线程

  • 创建用于强制清理不可抢占区域的“粗鲁”task RCU 线程；

  • 创建用于管理可睡眠 BPF 程序的内存回收的线程

  • 多核 CPU（SMP）启动前最高优先级初始化函数（initcall）的调用。后续会详细介绍initcall机制。
  • 通过函数do_one_initcall()执行位于__initcall_start和__initcall0_start之间的初始化函数。最高级别的初始化函数通常标注为early_initcall或通过 early_initcall 注册，它们位于__initcall_start和__initcall0_start之间。do_one_initcall()执行单个初始化任务。
  • 死锁及挂起检测机制初始化，主要完成：

  • 为每个 CPU 创建一个名为 watchdog/X]的高优先级内核线程，定期更新“喂狗”时间戳。

  • 利用硬件性能计数器产生 NMI 中断。由于 NMI 优先级极高，即使系统关中断，NMI 依然能触发并强制打印堆栈（Panic 或 Warn）。

函数smp_init()的作用是启动系统中除了引导 CPU (BSP) 之外的其他所有处理器内核，确保所有 CPU核芯都进入就绪状态，并建立多处理器之间的数据结构。 主要完成：

  • 通过idle_threads_init()为每个从内核（Secondary CPUs）创建 idle 线程

  • 利用cpuhp_threads_init()初始化 CPU 热插拔状态机线程

  • 调用bringup_nonboot_cpus()，通过 IPI（核间中断）或其它相关机制唤醒剩余内核

  • 通过smp_cpus_done()完成最后与体系结构相关的设置（如 x86 的 MTRR 设置或缓存同步）

Linux提供了两个版本的sched_init_smp()函数，分别用于对称多核系统和非对称多核系统，通过编译开关CONFIG_SMP选择。非对称版主要用于老式版，作用是初始化调度粒度。

现代版Linux大多采用对称版，其功能远复杂于非对称版。对称版sched_init_smp()函数位于git\kernel/sched/core.c，其主要功能包括：

  • 通过sched_init_numa()感知内存布局

初始化 NUMA拓扑，确保调度器知道离各个 CPU 最近的内存节点，从而实现就近分配

  • 通过sched_init_domains()构建调度域

根据 cpu_active_mask（当前活跃的 CPU 列表）建立由SMT 层、MC 层、NUMA 层构成的层次化调度树。SMT 层实现超线程间的负载均衡，MC 层实现物理内核间的负载均衡，NUMA 层实现跨节点间的负载均衡

  • 利用set_cpus_allowed_ptr()改变任务的CPU掩码

将当前的 init 进程从可能被隔离的核迁移到管家核上。这样可以确保启动后的核心系统进程不会占用那些专门预留给实时任务或特定高性能任务的核

  • 通过函数init_sched_rt_class()和init_sched_dl_class()初始化实时（Real-Time）调度类和Deadline 调度类

  • 设置状态标志位sched_smp_initialized为true

sched_smp_initialized是一个全局标志位，其值为 true标志着系统从单核调度器切换到了多核负载均衡调度器。

函数padata_init()、page_alloc_init_late()和page_ext_init()是内存管理系统的收尾。之前的内存管理初始化模块的核心任务是建立分配器本身，现在的三个函数基于内核已切换到多核模式，主要目的是建立辅助框架和功能扩展，以优化性能和初始化延迟。例如，在内存巨大的机器上，为了加快启动速度，内核会延迟初始化部分内存页，这时会完成延迟的内存初始化。系统还会根据当前的 CPU 拓扑精细化调整每个 CPU 的内存缓存（PCP）。

pdata_init()的功能是在为多核并行处理铺路，重点处理 CPU 热插拔和工作队列的动态预分配。

  • 注册回调函数 padata_cpu_online和 padata_cpu_dead

早期的内存初始化只管当前的 CPU，而 padata 必须考虑未来。如果系统运行中增加或减少了 CPU，并行任务的分发逻辑必须实时更新，否则会导致任务发送到不存在的核上。

  • 资源预分配 (kmalloc_array)

它根据 num_possible_cpus()一次性申请了一组 padata_work 结构。

  • 空闲链表管理 (padata_free_works)

将分配好的工作对象挂载到 padata_free_works 链表中备用。

这是一种典型的“池化”思想，为了在后续高频的加密/网络处理中快速获取对象，避免频繁分配内存带来的开销。

page_alloc_init_late()的主要功能是初始化被延迟初始化的内存，弃用早期使用的memory block内存分配算法。主要工作包括：

  • 完成推迟的初始化工作

在启动最早期，内核必须初始化所有物理页的 struct page 结构。这个过程极其耗时，会导致开机黑屏很久。内核在早期只初始化启动必需的一小部分内存，而在这里调用 kthread_run 为每个内存节点创建一个入口函数为deferred_init_memmap、名为pgdatinit 内核线程。利用多核优势同时初始化剩余的内存页表。 wait_for_completion 可以确保所有内存页都准备好后，才允许系统继续往后运行。

  • 通过函数zone_pcp_update()更新每个内存区域的PCP 批量分配限额

由于延迟初始化完成后，受管理的内存页总数发生了变化，这个函数会更新每个内存区域的 PCP (Per-CPU Pages) 批量分配限额。如果不做这一步，系统的内存分配性能会因为限额太小而变得非常低下。

  • 通过静态跳转技术直接从机器码层面删除内存分配路径中的检查延迟页逻辑，从启动模式切换到高性能运行模式。

  • 利用memblock_discard()弃用早期分配器

内核启动初期使用 memblock 算法管理内存，这是一种临时的、功能简单的分配方式。内核会将 memblock 占用的私有内存释放回伙伴系统。从此以后，系统完全交由正式的伙伴系统接管。

  • 通过shuffle_free_memory()将空闲内存页的顺序打乱

这样可以降低内存分配器的可预测性，增加内存安全性与随机化，防止针对内存布局的攻击，并减少某些特定负载下的缓存冲突。

  • 函数set_zone_contiguous的作用是标记内存区域的连续性状态，优化后续的大块内存分配。

函数page_ext_init()通过编译开关CONFIG_SPARSEMEM启用，也就是说只有内存模型为稀疏内存模型时才会启用，其余情况下函数为空函数。其主要作用是：

  • 通过invoke_need_callbacks()检查page_ext_ops中有哪些子系统想给每个物理内存页面描述符额外挂一小块私有扩展数据，以及它们各自怎么申请和初始化。如果没有开启任何需要页扩展的功能，函数会直接退出。

  • 通过遍历每个有内存的内存节点的各个段（内核将物理内存划分为段，通常为 128MB 或更小），检查物理页框（PFN）和节点的合法性（在某些高性能架构上，物理地址并不总是连续地属于同一个节点），为每个节点分配存储页扩展数据（page_ext结构体）的内存数组。页扩展数据是按段分配的。

  • 调用invoke_init_callbacks()实现动态挂钩，通知各个功能模块存储区已就绪。page_ext 本身只是提供了一个存储空间。

由于初始化大量页扩展数据（TB级内存）可能需要数秒时间，为了不让系统“假死”，在处理每个段的循环中调用了 cond_resched()，主动释放 CPU 权限，让其他重要的内核任务（如某些监控线程）有机会运行。

page_alloc_init_late()运行结束之后，核心调度器、内存管理和中断系统已经完全就绪，接下来的主要工作就是初始化大量外设。do_basic_setup() 是从核心子系统初始化转向设备和驱动程序初始化的关键节点。

如我们前面提到的，目前为止，内核启动只是初始化核心调度器、内存管理和中断系统等子系统，并没有触及外设及其驱动程序。从现在开始，kernel_init_freeable()通过调用do_basic_setup()搭建设备管理框架、划定资源边界、准备交互接口、以及大量的驱动加载。do_basic_setup()位于git/init/main.c，其定义为：

static void __init do_basic_setup(void)
{
	cpuset_init_smp();
	driver_init();
	init_irq_proc();
	do_ctors();
	usermodehelper_enable();
	do_initcalls();
}

函数cpuset_init_smp()的作用是在多核环境下初始化 cpuset 子系统，确保后续启动的进程（包括用户态的 init）可以根据配置被限制在特定的 CPU核芯或内存节点上运行。其核心任务是定义根 cpuset（即 top_cpuset）的资源边界。主要任务包括：

  • 通过掩码拷贝函数cpumask_copy()，把包括当前所有激活CPU的CPU掩码（cpu_active_mask）赋给根cpuset的允许CPU掩码（top_cpuset的cpus_allowed字段），将当前系统中所有处于激活状态的 CPU 核芯全部分配给根 cpuset。这保证了初始状态下，进程可以在任何 CPU 上运行。

  • 把所有包含内存的节点分配给根cpuset（top_cpuset的mems_allowed字段）管理

  • 把所有激活CPU的CPU掩码拷贝到根cpuset的有效资源CPU掩码（effective_cpus字段）

这里确保实际生效的掩码与配置掩码一致。

  • 注册内存热插拔通知函数cpuset_track_online_nodes_nb()

确保如果在系统运行时动态插入或拔出了内存条时cpuset 能接到通知，以便实时更新可用的内存节点列表。

  • 创建内存迁移工作队列

这样当用户手动更改某个 cpuset 的内存节点限制时，内核通过工作队列异步地将进程的内存页从旧节点迁移到新节点。

函数driver_init()的作用是初始化内核的设备驱动模型（Device Model），创建 sysfs 中的基础目录结构（如 /sys/devices, /sys/bus, /sys/class）。如果没有这一步，后续 do_initcalls 里的驱动程序就没地方挂载自己的设备节点，也无法实现即插即用。Linux 设备模型最终会在 sysfs 中表现为几大目录：

/sys/

├── devices/ 按物理/拓扑层级组织设备

├── bus/ 按总线组织 device / driver，其下包含devices和drivers等子目录

├── class/ 按功能分类组织设备

├── firmware/ 固件相关

├── hypervisor/ 虚拟化相关

└── dev/ 设备号到设备对象的映射

driver_init()本身不去注册具体驱动，而是先把后面所有设备 、总线 、及驱动能依附的基础框架准备好。

Linux 设备基础框架包括设备模型及配套基础设施。设备模型的核心对象包括：

  • 设备（device）

  • 驱动程序（driver）

  • 总线（bus）

  • 类（class）

而 firmware、sysfs（如 /sys/devices、/sys/bus、/sys/class）、devtmpfs 等，则是围绕设备模型运转的配套基础设施和用户空间接口。

driver_init()分成核心骨架（core pieces）和设备类型 /及平台层补充两组。完成核心骨架初始化的函数有：

  • devtmpfs_init();

  • devices_init();

  • buses_init();

  • classes_init();

  • firmware_init();

  • hypervisor_init();

这一组用以建立最底层、最基础的设备模型通用框架。

设备类型 /及平台层补充初始化函数依赖第一组，它们包括：

  • of_core_init();

  • platform_bus_init();

  • cpu_dev_init();

  • memory_dev_init();

  • container_dev_init();

这一组是在总框架建立后，再把一些具体的重要设备和总线类别加到框架里面。

函数devtmpfs_init()的功能是初始化设备临时文件系统（devtmpfs）。devtmpfs 是挂载在 /dev 背后的一个特殊文件系统，用来自动创建设备节点，这样用户空间程序通过访问设备节点访问设备及其驱动程序。设备节点为用户空间提供了访问内核设备和驱动的入口。

设备模型中的设备对象最终常常不仅要出现在 /sys 里，还要在 /dev 里有设备节点，而 devtmpfs_init() 就是在为设备对象以后如何映射成 /dev/xxx 节点打基础。

devices_init()初始化设备子系统的核心结构。它会建立设备在 sysfs 中的基础目录结构/sys/devices。设备对象最终会挂/sys/devices下面。本质上它是在搭建所有设备对象统一放置的框架，是真实设备层级视角的总根。后面会看到的/sys/devices/system/cpu/、/sys/devices/system/memory/、/sys/devices/platform/等都属于 /sys/devices 这棵树。

buses_init()初始化总线子系统，创建/sys/bus目录，以便按总线类型组织设备。从设备挂连接这个角度看，它为设备和驱动程序提供统一管理框架，给设备和驱动程序提供匹配与组织的公共场所。后续各种包括PCI、USB、platform、I2C和SPI等各类总线都依赖它。这些总线在 Linux 设备模型中都要通过统一的 bus_type 管理。

每种总线（比如 PCI、platform、USB）注册时，都会形成自己的 bus 子系统，如：

  • /sys/bus/pci/

  • /sys/bus/usb/

  • /sys/bus/platform/

  • /sys/bus/i2c/

每个 bus 通常在 sysfs 下会有devices和drivers两个子目录，如platform总线下的两个目录为

/sys/bus/platform/ devices和/sys/bus/platform/ devices/ drivers。这是 Linux 设备和驱动匹配的核心组织方式之一。

classes_init()初始化分类（class ）子系统。 Linux 设备模型按功能进行分类，有块设备（block）、网络设备（net）、终端设备（tty）等不同类别。它为设备提供从用户功能角度对设备分类管理的框架，建立/sys/class目录。一个设备可以同时在bus和class两个子目录存在，例如一个网卡设备可能通过PCI 总线连接，出现在 /sys/bus/pci，在功能上它属于网络设备，出现在 /sys/class/net。classes_init() 本质上是在建立给用户看设备功能分类的那棵树。

firmware_init()初始化固件子系统，给后续设备加载固件提供统一接口。它建立/sys/firmware，用于存储和固件相关系统信息、固件加载接口、平台固件对象（如 EFI、DMI、ACPI 某些接口）等。

hypervisor_init()初始化虚拟机监视器（hypervisor）相关设备模型接口，用于支持虚拟机环境中的一些抽象设备或虚拟机监视器相关对象，补充虚拟化环境下的设备模型入口。在虚拟机环境下，某些设备/接口可能会通过这里接入设备模型。它建立/sys/hypervisor，以承载虚拟化环境相关对象，本质上是在 sysfs 里给虚拟化宿主环境信息留一个标准入口。

of_core_init()初始化设备树核心支持，给设备树描述的硬件对象建立内核中的支撑结构。它是在为把设备树里的硬件描述接入 Linux 设备模型做准备。其作用本质就是把设备树中的硬件描述接入 Linux 设备模型体系，让后续 platform 设备注册、驱动匹配能顺利进行。

platform_bus_init()初始化并注册平台总线（platform bus），从而为 platform_device 和 platform_driver 提供统一的组织、匹配和管理框架，它是在建立非可枚举板载设备的统一挂载总线。平台总线是嵌入式 / SoC 平台里最重要的总线之一，在sysfs 中通常对应/sys/bus/platform和/sys/devices/platform。平台设备是指在 SoC 地址空间中的设备，通常集成在SoC芯片中，比如ARM架构里的UART、GPIO、RTC、SPI 控制器、I2C 控制器等。

cpu_dev_init()初始化 CPU 设备对象，并形成类似/sys/devices/system/cpu/的子目录，把 CPU 也纳入设备模型中，使其在 sysfs 中以设备形式出现并可管理。让CPU 成为 Linux 设备模型中的一个设备类对象，这样后续 CPU 热插拔、拓扑、频率管理等都更容易统一管理。

memory_dev_init()初始化内存设备对象，并创建/sys/devices/system/memory/目录。把内存块等内容接入设备模型，用于支持内存热插拔、内存块状态管理、用户空间查看物理内存布局等。它是在让物理内存也能以设备对象方式被统一管理。

container_dev_init()初始化容器设备对象（container device）。容器是设备模型里的一个逻辑容器对象，用于把某些设备组织在更高层的父对象下，常用于系统设备层次组织。

总之，driver_init() 不是初始化某个驱动，而是在内核早期把 Linux 统一设备模型的核心框架全部搭起来，为后续所有设备注册、驱动匹配、sysfs 展示和设备节点创建打基础。也就是“先把以后所有设备该怎么挂、怎么分组、怎么匹配、怎么在 sysfs 展示”这一整套规则搭起来。

函数init_irq_proc()的主要功能是/proc 目录下建立 irq 子目录。函数位于git/kernel/irq/proc.c，完成的工作包括：

  • 调用 proc_mkdir() 在 /proc 下创建 irq 文件夹

  • 执行 register_default_affinity_proc()，

在 /proc/irq/ 目录下创建一个名为 default_smp_affinity 的文件，绑定与该文件关联的回调函数，这允许用户查看或设置系统中新分配中断的默认 CPU 绑定策略

  • 遍历现有中断，调用 register_irq_proc(irq, desc)

在 /proc/irq/ 下为每个中断号创建一个子目录（如 /proc/irq/44/），并在其中生成smp_affinity 和 node 等中断管理接口和控制文件。

do_ctors()的作用是执行地址存在__ctors_start 和__ctors_end 之间的C++ 风格构造函数。虽然内核主要用 C 语言，但 GCC 编译器支持在 C 中使用 __attribute__((constructor)) 属性。在链接阶段，链接器将这些构造函数的地址存到__ctors_start 和__ctors_end 段。do_ctors() 遍历并运行由该属性标记的初始化函数，初始化一些C++风格的静态变量。只有通过编译开关CONFIG_CONSTRUCTORS启用，该函数才会执行这些构成函数。若关闭，函数为空函数。

usermodehelper_enable()的作用是开启内核与用户态的通道，启用UMH反向调用机制。UMH（usermodehelper）是 Linux 内核中一套允许内核空间直接启动并执行用户空间程序的框架。它实现了从内核到用户态的反向调用，是内核与用户态交互的重要机制。当驱动程序需要调用用户态程序（如模块自动加载、热插拔处理、或者固件加载）时，这个机制必须处于开启状态。

这里需要说明的是该函数只是把全局变量usermodehelper_disabled设置为UMH_ENABLED，当内核调用用户态的程序时，需通过call_usermodehelper()启用UMH反向调用机制。

后面的小节我们将介绍do_basic_setup()函数最后的、也是最重要的工作，即do_initcalls()函数。它通过initcall机制，按照 0-7 的级别顺序，把所有编译进内核的驱动（包括 rootfs）全部跑一遍。为了更好理解程序运作，下一节我们先介绍initcall工作机制。
