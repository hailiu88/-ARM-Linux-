基于ARM的Linux操作系统引导详解

内存初始化函数

内存初始化函数为mm_init()，定义在文件git/init/main.c文件里。其定义为：

static void __init mm_init(void)
{
	page_ext_init_flatmem();
	init_debug_pagealloc();
	report_meminit();
	mem_init();
	kmem_cache_init();
	kmemleak_init();
	pgtable_init();
	debug_objects_mem_init();
	vmalloc_init();
	ioremap_huge_init();
	init_espfix_bsp();
	pti_init();
}

函数page_ext_init_flatmem()的作用是为页面扩展分配一些内存空间。如果内存模型为稀疏模型，该函数为空操作，如果内存模型为平坦内存或不连续内存模型，该函数通过调用函数alloc_node_page_ext()，而alloc_node_page_ext()通过引导过程内存分配器（memblock_alloc_try_nid()）为每个内存节点分配一些连续内存。

当系统设置了CONFIG_DEBUG_PAGEALLOC和 _debug_pagealloc_enabled_early后，函数init_debug_pagealloc()的作用是通过设置跳转标号_debug_pagealloc_enabled和debug_guardpage_enabled的enabled字段的值，激活内存页面监控及防御页特性。如果内存页面监控被激活，当一个页面被释放回 伙伴系统 时，内核会通过修改页表项将其标记为“不可访问”。如果此时有任何代码尝试访问或越界写入这个已释放的页面，CPU 会立即触发 Kernel Panic。如果防御页特性被激活，内核会在特定的内存分配（通常是 vmalloc 或特殊的堆栈分配）之间插入一个不可访问的空隙页（Guard Page）。如果代码发生了顺序写入溢出，一旦触碰到这个“防御页”，CPU 会因为页表权限问题立即触发 Page Fault。它能让内核在损坏发生的一刻直接生成 Oops，而不是等到后续系统崩溃时才发现数据被污染了。

report_meminit()作用是在引导期间显示自动变量、堆、栈初始化时的值。自动变量（auto）在初始化时可以为全零，栈在初始化时可以为全零，结构体可以采用最强保护模式，也可以采用中等强度保护模式或最弱保护模式，堆可以在分配时全部赋值为零、也可以在释放时全部清零。该函数会生成类似于如下格式的日志：

mem auto-init: stack:all(zero), heap alloc:on, heap free:off

mem_init()函数定义在git/arc/arm/mm/init.c文件中，其的作用是在mem_map中标识到目前为止，系统占用后所剩自由空间的位置。该函数主要完成如下工作：

  • 利用swiotlb_init()、通过快内存分配器为SWIOTLB模块保留回弹缓冲区并初始化回弹缓冲所使用的数据结构

  • set_max_mapnr()函数设置系统最大页码（max_mapnr）

  • 利用函数free_unused_memmap()释放不再使用的内存映射（mmap所占的部分区域）

  • 利用函数memblock_free_all()把所有节点的结构体zone的managed_pages清零，并把由设备树标记为保留内存的物理页面标记为PG_reserved（不可用内存），然后将所有的自由内存从保留状态转换为自由状态（即清除PG_reserved标识）。把页面引用次数置为1，最后由_free_pages()函数将自由页面交给伙伴内存分配系统管理

  • 把步骤4释放的内存添加到全局计数器_totalram_pages上

  • 利用函数free_highpages()将所有高端自由内存释放給伙伴系统，并更新计数器totalhigh_pages 和 totalram_page

  • 利用函数mem_init_print_info()生成内存资源统计日志

kmem_cache_init()函数的作用是初始化内核内存分配缓冲区。在前面的介绍中我们可以发现，slab和slub内存分配器通过kmem_cache结构体分配大小低于一页的内存模块，因此系统必须初始化kmem_cache。Linux提供了分别用于slab和slub分配器的初始化模块，定义于git/mm/slab.c和git/mm/slub.c文件里。

引导阶段调用哪一个函数取决于编译配置开关。若设置了CONFIG_SLAB，则调用slab.c中的函数，若配置了CONFIG_SLUB，则调用slub.c里的函数。内核通过检查git/mm/Makefie里的obj-y确定调用哪一个函数。从前面的介绍我们可知，slab分配器利用结构体kmem_cache_node进行内存分配，因此，slab分配器的kmem_cache_init()首要工作是初始化kmem_cache_node结构体。对slab分配器而言，kmem_cache_init()的主要工作包括：

  • 建立三个slab在内存分配过程中必需的空链表slab_full、slab_partial和slab_free，并将kmem_cache_node的其它字段清零

  • 设置slab使用的全局变量slab_max_order

  • 创建一个类型为kmem_cache的引导缓冲区kmem_cache_boot

  • 将创建的kmem_cache链接到slab_cache链表

  • 利用create_kmalloc_cache ()函数创建用于存储kmem_cache_node结构体的缓冲区

  • 把init_kmem_cache_node的内容复制到kmem_cache和kmalloc_caches，期间会利用函数kmalloc_node()分配必要的内存

  • 如果数组单元不存在，利用函数new_kmalloc_cache()创建kmalloc_caches[type][index]二维数组中的各个单元。kmalloc_cache[type][index]供内存分配器在slab_cache中确定特定类型和大小的内存模块

kmemleak_init()初始化内存泄漏自动检测器，主要工作包括：

  • 设置分配内存后内存泄漏检测开始的最早时间jiffies_min_age。内存分配后，通常需要一点时间来建立指向该内存的指针，如果过早开始检测，由于指针可能还没写回内存，kmemleak 会误认为该内存已泄露。只有当一个内存对象的“存活时间”超过时，kmemleak 才会将其纳入“疑似泄漏”的扫描和报告范围

  • 设置后台线程自动扫描内存泄漏的间隔jiffies_scan_wait

  • 创建不需要泄漏检测线程跟踪的缓冲区object_cache，类型为结构体kmemleak_object，用于保存所有分配内存的元数据。检测线程需要利用这些元数据进行泄漏检测

  • 创建不需要检测线程跟踪的缓冲区scan_area_cache ，类型为结构体kmemleak_scan_area，用于保存内存模块内需要扫描的区域。

  • 生成对应于.data、.bss和ro_after_init段的元数据，并将这些段标识为灰色。在内存泄漏检测中，将内存状态定义为黑色、灰色和白色三种状态。黑色表示内存段绝对活跃，不需要扫描。灰色表示内存段绝对活跃，但需要扫描其内部，因为其内部可能包含指向其它内存模块的指针。白色表示状态待定，如果扫描结束时状态仍为白色，则可判断内存已发生泄漏。

pgtable_init()函数调用ptlock_cache_init()和pgtable_cache_init()函数，初始化用于页表锁的缓存区及页表专用的缓存区。

在多核系统中，为了避免资源竞争及进程同步，Linux使用大量不同类型的锁。ptlock_cache_init()函数创建名为 page->ptl的kmem_cache缓存区，用于高效分配和管理这些页表锁。缓冲区以kmem_cache结构体类型组织，地址由指针page_ptl_cachep指定。

pgtable_cache_init()函数初始化与硬件有关的页面缓冲区，对于64位的arm，当全局页目录（PGD–page global directory）大小与页面大小不同时，会分配一个名为pgd_cache的kmem_cache缓冲区，对于32位arm，该函数为空函数。

debug_objects_mem_init()函数是用于准备对象生命周期调试子系统（debugobjects）的函数。其核心任务是建立一个名为debug_objects_cache的内存池。该内存池保存所有用于存放被跟踪对象元数据的debug obj结构体。debug_objects_mem_init()还为每个CPU创建一个用于保存自己可用的degbug_obj哈希表。为了减少资源竞争，每个CPU还有自己的小内存池，存储部分debug_obj，供自己使用。

vmalloc_init(void)函数为从物理内存管理到虚拟内存管理转换做准备，是调用函数vmalloc()、vmap()和ioremap()的基础。它的主要任务是设置vmalloc()函数使用的虚拟内存空间。vmalloc_init()的工作主要包括：

  • 创建名为vmap_area、类型为结构体vmap_area的缓冲区。缓冲区的地址指针为vmap_area_cachep

  • 初始化每个CPU的vmap_block_queue结构体的字段值

为了降低内核之间在内存分配时的竞争程度，Linux为每个CPU分配少量类型为结构体vmap_block的专用内存模块，用以记录内核使用vmap_area的元数据。其中包括自旋锁、对应所用vmap_area模块的指针、所用模块是否空闲、由空闲模块构成的链表等信息。vmap_block_queue包含自旋锁和空闲vmap_block链表头两个成员。

  • 初始化每个CPU用于延迟释放内存的vfree_deferred结构体

该结构体包含延迟释放链表头及描述实际释放进程的work_struct结构体。在许多情况下，进程使用的vmap_area不能立即释放。这时需要把要释放的内存模块放到vfree_defered指向的延迟释放列表中，之后由释放进程free_work()在合适的时机释放。

  • 把保存在vmlist中的所有vm_struct结构体转换为vmap_area使用的格式，转移到vmap_area，以便集中管理

  • 遍历记录在vmap_area_list链表中所有已分配的虚拟内存模块

通过遍历链表，确定vmap_area中的空闲模块，把所有空闲模块记录到以free_vmap_area_root为根的红黑树中，并将各个虚拟内存模块vmap_area的list字段（用以链接成链表）添加到表头为free_vmap_area_list的链表中。

ioremap_huge_init()函数用于初始化决定是否在i/o映射中使用大页（PGD支持2MB，PUD支持1GB）的标志位。这些标志位分别为ioremap_p4d_capable，ioremap_pud_capable和ioremap_pmd_capable，是否支持与硬件架构有关。

init_espfix_bsp()函数是Linux内核为修复因特尔公司x86_64架构的ESP 栈指针漏洞的补丁，对arm而言，该函数为空函数。

pti_init()是mm_init()函数调用的最后一个子程序。pti_init()是初始化 KPTI（内核页表隔离） 机制的核心函数，是为了应对熔断（meltdown）处理器漏洞而引入。该函数的核心是限制用户空间的内存映射。当进程在用户空间运行时，只映射极少数必要的内核地址。只有当进程返回内核空间运行，系统才会恢复正常的地址映射。
