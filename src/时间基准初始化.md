## 时间基准初始化

在安全机制建立之后，内核开始从基础环境建设向功能性系统过渡。紧随
rand_initialize() 和 boot_init_stack_canary()
之后的四个函数标志着内核开始建立其时间度量、性能监控和多核通信能力。下面先介绍内核时间基准的建立过程。

time_init()
是内核校准硬件时钟的关键点。它会调用架构相关的代码（如arm板级提供的init_time()函数、通过设备树时钟节点及通过TIMER_OF_DECLARE
宏定义提供的时钟）来初始化系统定时器。在这之前，内核的 jiffies
仅基于估算，此后，内核才真正有了精确的“墙上时间。

time_init()定义在各架构目录下，对32位arm处理器而言，函数定义在文件
arch/arm/kernel/time.c中，其定义为：

```
void __init time_init(void) 
{
	if (machine_desc->init_time) {
		machine_desc->init_time();
	} else {
#ifdef CONFIG_COMMON_CLK
		of_clk_init(NULL);
#endif
		timer_probe();
		tick_setup_hrtimer_broadcast();
	}
}
```
该函数通过一个 if-else 结构兼容了传统 Machine
描述符模式和现代设备树与通用时钟模式两种硬件初始化模式。如果当前板级文件定义了
init_time 回调函数，则直接调用它。这种情况多见于旧款 SoC
或未完全迁移到设备树（Device Tree）的 ARM
设备。硬件厂商会在板级代码中手动设置定时器频率和中断。如果系统启用了通用时钟框架（CONFIG_COMMON_CLK），则通过函数of_clk_init()，根据
设备树
节点初始化整个系统的时钟树。of_clk_init(NULL)从设备树的根节点开始，遍历整个设备树中所有带有时钟相关属性的节点。一旦发现设备树节点的
compatible 属性与在由 CLK_OF_DECLARE
宏（时钟宏，用于注册时钟控制器驱动函数）声明的匹配表中的某个时钟驱动匹配，它会执行该驱动提供的初始化回调函数。CLK_OF_DECLARE
宏用于在内核镜像中注册时钟驱动。如果设备树上的时钟呈现层级关系，of_clk_init()会按层级关系初始化系统时钟。

在 time_init() 中，timer_probe() 紧随 of_clk_init()
之后执行，是内核接管硬件定时器的最后临门一脚。

如果说 of_clk_init() 是把时钟源找出来，那么 timer_probe()
就是把闹钟调好并上弦。在 Linux 设备驱动模型中，probe()
函数是驱动和设备成功匹配后执行的核心初始化函数。当内核发现某个设备与驱动的
id_table / compatible 匹配时，就会调用驱动的 probe()。简单说，probe()
的主要任务是完成设备的初始化并让设备可以正常工作。

timer_probe() 定义在 drivers/clocksource/timer-probe.c
中。它的主要任务是匹配定时器硬件、调用定时器驱动以及建立基准。timer_probe()首先扫描设备树中带有
compatible 属性的定时器节点，在由 TIMER_OF_DECLARE
宏（定时器宏，用于注册定时器驱动程序）注册的匹配表中查找与compatible
属性匹配的字符串。如果发现匹配的节点，则执行对应定时器驱动的初始化函数。之后，内核就可以确定时钟源硬件和
时钟事件设备。

of_clk_init的作用是准备通用时钟框架，通过它初始化 SoC
的时钟树。该函数确保系统中的各个时钟节点（如
PLL、分频器、门控时钟）都已就绪。定时器驱动在初始化时，通常需要获取一个输入频率才能正常工作。它会调用时钟接口来获取由
of_clk_init() 提前注册好的时钟信号。

执行完 timer_probe()
后，物理定时器已开始计数，并能在计数到期时触发中断。至此，内核的时钟节拍机制已经准备就绪，内核才真正具备了处理多任务的时间分片能力。
