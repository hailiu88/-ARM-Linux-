## 准备进程运行资源

在 Linux 内核启动流程中，紧随 fork_init() 之后的 proc_caches_init()
是为进程运行所需的通用内核对象建立缓存区的初始化函数。函数位于git/kernel/fork.c，其定义为：

```
void __init proc_caches_init(void)
{
	unsigned int mm_size;
	sighand_cachep = kmem_cache_create("sighand_cache",
		sizeof(struct sighand_struct), 0,
		SLAB_HWCACHE_ALIGN|SLAB_PANIC|SLAB_TYPESAFE_BY_RCU|
		SLAB_ACCOUNT, sighand_ctor);
	signal_cachep = kmem_cache_create("signal_cache",
		sizeof(struct signal_struct), 0,
		SLAB_HWCACHE_ALIGN|SLAB_PANIC|SLAB_ACCOUNT, NULL);
	files_cachep = kmem_cache_create("files_cache",
		sizeof(struct files_struct), 0,
		SLAB_HWCACHE_ALIGN|SLAB_PANIC|SLAB_ACCOUNT, NULL);
		fs_cachep = kmem_cache_create("fs_cache",
		sizeof(struct fs_struct), 0,
		SLAB_HWCACHE_ALIGN|SLAB_PANIC|SLAB_ACCOUNT, NULL);
	mm_size = sizeof(struct mm_struct) + cpumask_size();
	mm_cachep = kmem_cache_create_usercopy("mm_struct",
		mm_size, ARCH_MIN_MMSTRUCT_ALIGN,
		SLAB_HWCACHE_ALIGN|SLAB_PANIC|SLAB_ACCOUNT,
		offsetof(struct mm_struct, saved_auxv),
		sizeof_field(struct mm_struct, saved_auxv), NULL);
	vm_area_cachep = KMEM_CACHE(vm_area_struct, SLAB_PANIC|SLAB_ACCOUNT);
	mmap_init();
	nsproxy_cache_init();
}
```

该函数主要调用 kmem_cache_create()
和kmem_cache_create_usercopy为信号处理描述符（<u>sighand_struct</u>）、信号描述符（signal_struct）、文件描述符（files_struct）、文件系统描述符（fs_struct）以及内存管理描述符（mm_struct）等关键结构创建名为sighand_cache、signal_cache、files_cache、fs_cache和mm_struct的缓冲区。这些描述符是进程运行和管理的基础，保存了进程运行过程中各种资源信息。由于这些结构的大小是固定的，通过
SLAB/SLUB 分配器 预先分配这些结构，可以避免在频繁 fork()
时反复从通用内存池中寻找零碎空间。专门的缓存池可以极大地减少物理内存的碎屑化。

在分配完所需的缓冲区后，proc_caches_init()调用宏KMEM_CACHE()，创建一个空的虚拟内存区域描述符（vm_area_struct），用于进程虚拟内存分配的模板。之后在创建进程时，就会按这样的模板分配虚拟内存。

mmap_init()的作用时初始化内核用于管理进程虚拟地址（VMA）的全局辅助数据结构。例如，它会处理系统默认的
mmap
基地址选择算法，并初始化一些与虚拟内存映射相关的内核通知机制（Notifiers）。与前面介绍的mm_init()函数不同，mm_init()的核心任务是初始化物理内存管理和基础内存分配器，如设置内存节点、内存区域、页表以及
伙伴系统。

最后，函数利用nsproxy_cache_init()建立 slab
缓存（对象分配池），供进程命名空间代理对象高效分配与释放。在现代 Linux
中，进程不只是跑在宿主机上，还可能跑在容器里。nsproxy
本身不是命名空间，它是 Linux
内核里一个命名空间集合代理对象。它聚合了主机名（UTS）、进程间通信（IPC）、网络（NET）、文件挂载（MNT）等
6 种以上的命名空间指针。由于很多进程共享相同的命名空间组合（比如同一个
Docker 容器里的所有进程），通过 nsproxy
缓存和引用计数，内核可以极大地减少重复创建命名空间对象的开销。

这部分初始化完成后，Linux
内核就已经具备了精细化管理进程内存和多维度隔离进程环境的能力。
