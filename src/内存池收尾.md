## 内存池收尾

在 SLAB/SLUB
分配器基本就绪后，还有些内存池的初始化处理依赖于中断或特定延迟。在中断和时钟系统初始化后，需要把内存池初始化那些依赖中断或特定延迟的操作完成。kmem_cache_init_late()主要负责完成那些之前无法进行的初始化任务。

Linux定义了三个同样名字的函数，分别该位于文件git/mm/slab.c、git/mm/slub.c和git/mm/slob.c，start_kernel()，调用哪个版本取决于编译配置。如果定义了CONFIG_SLUB（现代默认），则调用git/mm/slub.c中的函数kmem_cache_init_late()。如果定义了CONFIG_SLAB，则调用
mm/slab.c中的函数。如果定义了 CONFIG_SLOB，则调用
mm/slob.c中的函数(多数情况下为空函数)。slob.c中的函数主要用于嵌入式微小型系统，slob算法在
Linux v6.4
版本中已被正式移除。下面我们以slab.c中的函数为例介绍kmem_cache_init_late()的主要工作。函数的定义为：

```
void __init kmem_cache_init_late(void)
{
	struct kmem_cache *cachep;
	mutex_lock(&slab_mutex);
	list_for_each_entry(cachep, &slab_caches, list)
		if (enable_cpucache(cachep, GFP_NOWAIT))
			BUG();
	mutex_unlock(&slab_mutex);
	slab_state = FULL;
#ifdef CONFIG_NUMA
	hotplug_memory_notifier(slab_memory_callback, SLAB_CALLBACK_PRI);
#endif
}
```

该函数首先通过list_for_each_entry()遍历全局链表
slab_caches中的所有已创建的缓存，并通过函数enable_cpucache()激活 CPU
缓存。这会将每个缓存的本地队列（per-CPU
array）调整到最终大小（由系统的内存总量和 CPU
数量决定），以提升多核并发性能。如果调整失败，系统无法正常运行，直接触发
BUG()。

激活缓存区后，函数切换缓冲区状态，将全局变量 slab_state 设置为
FULL。这意味着内存分配器已经完全进入工作状态。此时，内核可以放心使用涉及
kmalloc 的复杂操作，包括一些依赖完整分配器功能的调试工具和监控功能。

如果开启了
NUMA，它会注册一个内存热插拔通知器。当系统增加或移除物理内存条时，slab能够得到通知，从而动态地调整每个节点（Node）的内存管理数据结构。

这段代码是slab分配器的最后一道通电工序。它把缓存队列扩容到最大，并开启了对动态内存硬件的支持
