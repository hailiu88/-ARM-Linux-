## 工作队列早期初始化

这是工作队列初始化的前半段。在 Linux 内核中，workqueue_init_early() 是
工作队列子系统的核心初始化函数。它在系统启动的早期被调用，负责在系统具备创建线程能力之前，搭建起工作队列的基础骨架。为了更好地理解工作队列的初始化过程，下面简单介绍一下Linux一个任务（由work结构体描述）的处理过程。

在 Linux 内核中，工作队列（Workqueue ）
是一种将任务推迟执行的异步机制，运行在进程上下文中。这与直接运行一个进程不同，工作队列是将任务（函数）委托给内核创建的一组特殊的执行者线程（Worker
Threads）来执行。在整个任务执行过程中，要利用三个非常重要的数据结构，它们决定了任务由谁处理、怎么处理。这三个数据结构分别是：

- 任务单（work_struct）

> 用户定义的最小任务单元。它包含一个函数指针（你要执行的具体逻辑）和数据。

- 工作者或执行者（worker）

> 内核中的一个线程（名字通常是
> kworker/uX:X），负责从队列里取任务并执行。

- 工作者池或执行者池(worker_pool)

> 管理一组 执行者。每个 CPU
> 通常有两个池，一个是普通优先级的，另一个是高优先级的。

在调用 schedule_work() 函数时，内核首先确定当前 CPU 对应的
执行者池，并将任务单 挂到
执行者池的worklist这个双向链表的末尾（pool-\>worklist
）。之后，内核通过池的空闲列表（pool-\>idle_list）检查池中是否有空闲的
执行者。如果有，则唤醒该 执行者的进程状态（从 S 变为
R）。如果没有空闲执行者且当前执行者都在忙，内核会根据负载决定是否创建一个新的
kworker。

执行者线程的执行过程是一个循环，它本质上是一个“领料-加工”的过程。执行者线程被唤醒后，进入
worker_thread() 函数。它会检查 pool-\>worklist
是否为空。如果不为空，它会取走链表头部的第一个
任务单（work_struct）。在执行前，内核会通过 pool_workqueue ()
来确保任务在正确的环境下运行。执行者会标记自己处于
工作状态，并调用任务单指定的任务（即
work-\>func指定的函数）。此时，任务正式开始在内核态执行。

Linux 使用的是 CMWQ 机制。它的调度逻辑保持 CPU 繁忙但不过载。如果一个
执行者在执行任务时进入了睡眠（例如等待磁盘
I/O），内核会立刻检测到，并唤醒池中的另一个
执行者来处理队列中的下一个任务。同一个任务单绝不会在多个 CPU
上同时运行。

当 任务单指定的任务完成后（即work-\>func 执行完毕），执行者
会检查是否有其他任务需要处理。如果 执行者池的worklist 为空，该 执行者
不会被立即销毁，而是进入 pool-\>idle_list
睡眠。如果长时间没有新任务，该线程才会被内核回收。如果有进程在等待这个任务的完成，该执行者会被唤醒。

从用户提交任务到完成，本质上是任务被挂载到 CPU
的工作池子，唤醒一个内核线程，由这个线程代为执行。

workqueue_init_early()函数首先初始化工作队列所需的各种数据结构，确保内核在后续的初始化阶段（如驱动加载）能够正常分配和使用工作队列。在初始化全局
工作者池、工作队列所需链表以及各种锁后，设置 用于非绑定工作队列的 CPU
掩码用于标志不适合执行队列工作的CPU掩码
wq_unbound_cpumask。此外，该函数还为每个可能的
CPU创建并初始化两个默认的工作者池，一个普通优先级池和一个高优先级（highpri）池。最后为系统分配预定义的标准全局工作队列，例如
system_wq、system_highpri_wq、system_long_wq
等，以便内核其他组件直接使用。

函数workqueue_init_early()定义在文件qit/kernel/workqueue.c中，其定义为：

```
void __init workqueue_init_early(void)
{
	int std_nice[NR_STD_WORKER_POOLS] = { 0, HIGHPRI_NICE_LEVEL };
	int hk_flags = HK_FLAG_DOMAIN | HK_FLAG_WQ;
	int i, cpu;

	BUILD_BUG_ON(__alignof__(struct pool_workqueue) < __alignof__(long long));
	BUG_ON(!alloc_cpumask_var(&wq_unbound_cpumask, GFP_KERNEL));
	cpumask_copy(wq_unbound_cpumask, housekeeping_cpumask(hk_flags));
	pwq_cache = KMEM_CACHE(pool_workqueue, SLAB_PANIC);
	for_each_possible_cpu(cpu) {
		struct worker_pool *pool;
		i = 0;
		for_each_cpu_worker_pool(pool, cpu) {
			BUG_ON(init_worker_pool(pool));
			pool->cpu = cpu;
			cpumask_copy(pool->attrs->cpumask, cpumask_of(cpu));
			pool->attrs->nice = std_nice[i++];
			pool->node = cpu_to_node(cpu);
			mutex_lock(&wq_pool_mutex);
			BUG_ON(worker_pool_assign_id(pool));
			mutex_unlock(&wq_pool_mutex);
		}
	}
	for (i = 0; i < NR_STD_WORKER_POOLS; i++) {
		struct workqueue_attrs *attrs;
		BUG_ON(!(attrs = alloc_workqueue_attrs()));
		attrs->nice = std_nice[i];
		unbound_std_wq_attrs[i] = attrs;

		BUG_ON(!(attrs = alloc_workqueue_attrs()));
		attrs->nice = std_nice[i];
		attrs->no_numa = true;
		ordered_wq_attrs[i] = attrs;
	}
	system_wq = alloc_workqueue("events", 0, 0);
	system_highpri_wq = alloc_workqueue("events_highpri", WQ_HIGHPRI, 0);
	system_long_wq = alloc_workqueue("events_long", 0, 0);
	system_unbound_wq = alloc_workqueue("events_unbound", WQ_UNBOUND,
					    WQ_UNBOUND_MAX_ACTIVE);
	system_freezable_wq = alloc_workqueue("events_freezable",  WQ_FREEZABLE, 0);
	system_power_efficient_wq = alloc_workqueue("events_power_efficient",
					      WQ_POWER_EFFICIENT, 0);
	system_freezable_power_efficient_wq = alloc_workqueue("events_freezable_power_efficient",
					      WQ_FREEZABLE | WQ_POWER_EFFICIENT,    0);
	BUG_ON(!system_wq || !system_highpri_wq || !system_long_wq ||
	      !system_unbound_wq || !system_freezable_wq ||
	       !system_power_efficient_wq ||  !system_freezable_power_efficient_wq);
}
```

在开始介绍代码之前，先介绍两个预定义的常数和两个标志位：

- NR_STD_WORKER_POOLS

> 定义每个CPU拥有的工作者池的数量，通常为2。

- HIGHPRI_NICE_LEVEL

> 用以定义最高优先级修正值，通常为-20。优先级修正值越小，修正后的进程优先级越高。

- HK_FLAG_DOMAIN

> 管家调度域标志位。当这一标志位=位置1时，该 CPU 需要负责调度域相关的
> 杂物清理工作。

- HK_FLAG_WQ

> 管家工作队列标志位。当这一标志位置1时，该 CPU 负责执行
> 工作队列相关的后台工作。

第一行代码BUILD_BUG_ON(\_\_alignof\_\_(struct pool_workqueue) \<
\_\_alignof\_\_(long
long))的作用是保证结构体pool_workqueue八字节对齐。如果没有对齐，则在编译期间报错。第二行代码的作用是为cpu掩码变量wq_unbound_cpumask分配内存。如果分配内存失败，则出错报警。之后，初始化函数利用函数housekeeping_cpumask()获取cpu管家掩码housekeeping_mask，并通过函数cpumask_copy()把cpu管家掩码赋予wq_unbound_cpumask。这一行代码的意思就是位于不绑定到固定
CPU 的任务队列里的（unbound workqueue）
的任务只能在同时参与调度域并负责队列的 CPU 上运行。如果某 CPU 被
isolcpus 隔离，它就不会出现在
cpu管家掩码housekeeping_cpumask的第六位(HK_FLAG_WQ)。

pwq_cache =
KMEM_CACHE()的作用是为结构体pool_workqueue（简称pwq）在内核缓冲区分配内存，并把首地址赋予pwq_cache指针。如果分配内存失败，则产生内核崩溃错误。在
Linux 内核的 CMWQ (Concurrency Managed Workqueue) 机制中，pool_workqueue
是连接工作队列与底层线程池的核心中间层。它充当了面向用户的任务队列结构体（
workqueue_struct）与面向执行体的执行者池（ worker_pool）
之间的多对一映射表。一个工作队列（如 system_wq）在每个 CPU 或 NUMA
节点上都会对应一个 pwq，记录了特定工作队列在该
CPU/节点上的执行参数（如最大并发数 max_active）。

之后的循环模块初始化每个CPU的所有执行者池。在这段代码中，首先利用函数init_worker_pool()初始化池所用的链表和哈希表，并把其余各字段设为默认值，然后初始化结构体的cpu字段、NUMA节点字段以及attr字段的cpu掩码字段和优先级修正值字段。在该循环模块的最后通过函数worker_pool_assign_id()为每个池分配一个标号。

下一个循环模块是为每一个不与CPU绑定的工作队列和顺序执行的工作队列成员分配用于保存描述
执行属性 （运行在哪些 CPU、以什么优先级、属于哪个 NUMA
节点等属性）结构体的内存，并初始化这些队列的默认属性。所分配保存执行属性的内存地址记录在数组unbound_std_wq_attrs\[\]和ordered_wq_attrs\[\]中。

最后，workqueue_init_early()为一些通用的系统队列分配内。这些系统队列为：

- system_wq

> 系统默认提供的、最常用的 并发管理工作队列
> (CMWQ)。当不想自己创建一个专门的内核线程池，而只是想异步执行一个简单的函数时，它是首选方案。

- system_highpri_wq（系统高优先级工作队列）

> 是 Linux 内核预定义的一个特殊工作队列。它在底层结构上与 system_wq
> 相似，但在调度策略和响应速度上有本质区别。它的存在不是为了跑得快（那是
> system_highpri_wq
> 的活），而是为了别挡道。当任务预计会运行较长时间，或者包含可能导致长时间阻塞的操作（如复杂的磁盘
> I/O）时，就应该将其投入 system_long_wq。

- system_unbound_wq（系统不绑定工作队列）

> 是 Linux工作队列体系中最为特殊的一员。与之前提到的 system_wq 或
> system_highpri_wq 不同，它打破了“每个 CPU 一个池子”的束缚，专门为 CPU
> 密集型或长时间阻塞的任务设计。普通的 工作队列 是 Per-CPU
> 的，任务在哪个 CPU 上提交，就在哪个 CPU 的工人池里排队。而
> system_unbound_wq 标记为 WQ_UNBOUND，任务不与特定的 CPU
> 核绑定，可以在系统内任何允许的 CPU 上运行。虽然失去了 CPU
> 缓存热度（Cache
> Locality），但换来了更高的调度灵活性。它的存在主要是为了避免干扰关键任务和提高并行吞吐量。

- system_freezable_wq

> 是 Linux
> 内核中专门为支持电源管理（休眠/挂起）而设计的工作队列。当系统准备进入睡眠状态（Suspend/Hibernate）时，该队列中的任务会被冻结。在
> Linux 执行 suspend
> 过程中，内核必须确保所有可能干扰磁盘镜像写入或硬件状态切换的任务都停止运行。当内核启动
> Freeze 进程 时，该队列会停止派发新任务，并让现有执行者进入受控状态。

- system_power_efficient_wq

> 在 Linux 内核追求能效比的过程中，system_power_efficient_wq
> 扮演着节能调解员的角色。它的核心逻辑是在任务紧迫性与系统功耗之间寻找平衡。

system_power_efficient_wq
最独特的地方在于其行为取决于内核参数workqueue.power_efficient。默认状态（节能模式关闭）下，它等同于
system_wq。任务绑定在提交它的 CPU
上运行，以获得最佳的缓存局部性。节能模式下，它会变为 不绑定属性。这是
Linux 内核工作队列体系中的集大成者。system_freezable_power_efficient_wq
结合了 电源管理（冻结）与 能效优化（动态绑定）
两大特性。可以把它理解为一个既能省电，又能在休眠前停工的执行者。在内核初始化时，这个队列同时携带了WQ_FREEZABLE和WQ_POWER_EFFICIENT两个关键标志。任务在系统挂起时必须停止。它们倾向于在已经唤醒的
CPU 上运行，以减少 CPU
唤醒次数。这解决了背景轮询任务在移动设备上的省电和安全挂起两大难题。平时运行时，它尽量不惊动睡休眠的
CPU核芯。当系统关机或休眠时，它确保这些背景任务不会在硬件掉电的瞬间尝试访问设备，从而避免系统崩溃。

在初始化函数的末尾，利用宏BUG_ON()确保这些内存分配没有发生问题。
