## 函数运行跟踪子系统简介

函数跟踪子系统（ftrace，Function Tracer）是 Linux
内核内置的跟踪子系统。它能够以极低的开销监控几乎所有内核函数的调用过程。
函数跟踪子系统的高效源于其“动态跟踪”技术（Dynamic
Ftrace），它通过动态指令替换实现对函数的动态跟踪。动态指令替换的实现需要通过编译和初始化两个阶段的配合。

在内核编译时，如果如果开启了 CONFIG_FUNCTION_TRACER(老版Linux通过使用
GCC 的 -pg 选项)，在每个内核函数入口插入一个 mcount（或
\_\_fentry\_\_）调用指令。在初始化函数跟踪子系统（ftrace_init）时，初始化函数会识别出这些调用点，并将其指令全部替换为
NOP（空操作）。这样的操作系统性能开销几乎为零。当要跟踪时某个函数时，内核会用跳转到跟踪器（Trampoline）的指令（补丁）替换对应位置的
NOP 指令，所用技术与第12章介绍的跳转标号修改技术相同。

函数跟踪子系统可以跟踪的内容包括：

- 内核函数调用及其父函数

- 以树的形式展示函数调用关系，并记录执行耗时

- 跟踪并记录中断被禁用的最长时间及位置，用于分析系统延迟

- 跟踪禁止抢占的时间段

默认状态下，系统不启用跟踪操作。需要利用系统命令，通过位于tracefs文件系统的交互接口，启动函数跟踪。
现代操作系统一般会自动把tracefs文件系统通常挂接在/sys/kernel/tracing。

启用函数跟踪系统的流程一般为：

- 挂接tracefs文件系统（现代操作系统会自动挂接）

```
 mount -t tracefs nodev /sys/kernel/tracing
```

- 设置跟踪类型：

```
 echo function_graph \> /sys/kernel/tracing/current_tracer
```

- 设置过滤条件：

```
 echo 'vfs_read\*' \> /sys/kernel/tracing/set_ftrace_filter
```

- 启动跟踪：

```
 echo 1 \> /sys/kernel/tracing/tracing_on
``` 

跟踪结构存放在 /sys/kernel/tracing/trace。
