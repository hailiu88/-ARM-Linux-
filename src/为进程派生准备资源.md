## 为进程派生准备资源

在 Linux 中，进程派生（Forking）
是创建新进程的核心机制。它遵循复制-修改的哲学。进程派生不是从零开始创建一个空白进程，而是通过克隆父进程生成一个几乎完全一样的副本（子进程）。

在调用 fork()
系统调用时，内核为新产生的子进程分配一个唯一的进程标识符（PID），然后克隆父进程的进程控制块（task_struct），包括凭证、打开的文件描述符、环境变量等。为了节省内存，内核不会立即复制父进程的所有内存，而是与父进程共享同一块物理内存页。只有当其中一个进程尝试修改内存数据时，内核才会真正复制该内存页。

fork() 在父进程和子进程返回值不同。在父进程中返回子进程的
PID，在子进程中返回 0。父进程通过fork() 返回的 ID
来追踪和管理子进程，子进程通过fork()返回的0值判别自己是否是派生出来的副本。下面实例示范如何利用fork()的返回值区分父进程和子进程。

  ```
  #include <stdio.h>
#include <unistd.h>
#include <sys/types.h>
void fork_example() 
{
    pid_t pid = fork();
    if (pid < 0) {
        // fork 失败
        	perror("fork failed");
        	return;
    } 
    else if (pid == 0) {
        	// 子进程
        	printf("Child process: PID=%d, PPID=%d\n", getpid(), getppid());
    } 
    else {
        	// 父进程
        	printf("Parent process: PID=%d, Child PID=%d\n", getpid(), pid);
    }
}
```

在 Linux 中，派生通常只是第一步。完整的程序启动流程通常包括三步：

- 利用Fork创建一个完全一样的子进程。

- 子进程调用
  execve()。这会冲掉从父进程继承来的旧代码和数据，加载并运行一个新的二进制程序（如
  Nginx 或 LS）。

- 父进程通常会等待子进程结束，并回收其残留的资源，防止其变成“僵尸进程”。

父进程是母体，负责管理和监控。子进程是副本，继承了母体的身份（凭据）和资源（文件句柄），随后通过
exec 转型为具体的后台任务或程序实例。

在 Linux 内核源码（如 init/main.c）中，fork_init()
是内核启动过程中的一个关键初始化函数。它的核心任务是根据系统的物理内存容量，计算并设置系统允许创建的最大进程限制。函数位于git/kernel/fork.c，其定义为：

```
void __init fork_init(void)
{
	int i;
#ifndef CONFIG_ARCH_TASK_STRUCT_ALLOCATOR
#ifndef ARCH_MIN_TASKALIGN
#define ARCH_MIN_TASKALIGN	0
#endif	// ！ARCH_MIN_TASKALIGN
	int align = max_t(int, L1_CACHE_BYTES, ARCH_MIN_TASKALIGN);
	unsigned long useroffset, usersize;
	task_struct_whitelist(&useroffset, &usersize);
	task_struct_cachep = kmem_cache_create_usercopy("task_struct",
		arch_task_struct_size, align,
		SLAB_PANIC|SLAB_ACCOUNT,
		useroffset, usersize, NULL);
#endif		//！CONFIG_ARCH_TASK_STRUCT_ALLOCATOR
	arch_task_cache_init();
	set_max_threads(MAX_THREADS);
	init_task.signal->rlim[RLIMIT_NPROC].rlim_cur = max_threads/2;
	init_task.signal->rlim[RLIMIT_NPROC].rlim_max = max_threads/2;
	init_task.signal->rlim[RLIMIT_SIGPENDING] =
	init_task.signal->rlim[RLIMIT_NPROC];
	for (i = 0; i < UCOUNT_COUNTS; i++) {
		init_user_ns.ucount_max[i] = max_threads/2;
	}
#ifdef CONFIG_VMAP_STACK
	cpuhp_setup_state(CPUHP_BP_PREPARE_DYN, "fork:vm_stack_cache",
		  NULL, free_vm_stack_cache);
#endif		//CONFIG_VMAP_STACK
	scs_init();
	lockdep_init_task(&init_task);
	uprobes_init();
}
```

有些硬件架构有自己专用的进程描述符（task_struct
）分配器，因此fork_init()通过编译开关CONFIG_ARCH_TASK_STRUCT_ALLOCATOR，把初始化分为有无架构自定义
task_struct
分配器两种情况进行初始化。如果没有启用架构自定义的进程描述符分配器，则走通用分配路径。ARM32没有自己的分配器，在初始化时调用通用分配器。这里我们仅介绍无架构专有分配器时的函数工作。

函数首先计算 进程描述符的对齐大小，取
CPU缓存行大小和架构最小对齐中的较大值，保证缓存友好和架构要求，然后获取
task_struct 中用户拷贝白名单的偏移和大小（用于 hardened usercopy
安全机制）。用户拷贝白名单是允许被函数copy_to_user()拷贝到用户空间或用函数copy_from_user()把用户空间数据拷贝到内核空间的一小部分内存区域，这样做的目的是防止把内核敏感数据泄露给用户或从用户拷入非法数据破坏内核。

函数调用 kmem_cache_create() 为进程描述符创建一个专用的 SLAB/SLUB
缓存。这样以后每次创建新进程时，内核可以直接从这个预先分配好的缓存池里拿内存，速度极快。

对使用架构专有分配器的CPU，arch_task_cache_init()的作用是初始化架构特有的任务缓存区域。对使用通用分配器的CPU来说，这一函数为弱的空函数。

函数接下来的工作是为初始进程（init
进程）设置一系列上限，包括系统允许的最大线程/进程数（max_threads）、每个用户最大进程数的软上限和硬上限、每个用户最多可以排队的信号数量的软上限和硬上限。系统允许的最大线程/进程数设为MAX_THREADS，其余的软硬上限均设为max_threads/2。

遍历系统初始创建的最顶层的用户命名空间（init_user_ns），其下包括用户命名空间（user
namespace）、PID 命名空间（PID namespace）、UTS 命名空间（UTS
namespace）、IPC 命名空间（IPC 命名空间）、网络命名空间（network
namespace）、挂载命名空间等（mount
namespace），设置一个用户可以创建的各种命名空间数量上限。每个用户可以创建的各种命名空间的上限均为max_threads/2。

如果系统启用了VMAP_STACK（内核栈虚拟映射），则通过函数cpuhp_setup_state()注册
CPU 热插拔上线和下线的状态回调。上线回调函数为空，下线时的回调函数为
free_vm_stack_cache ，用以下线时释放栈缓存。

fork_init()
在最后调用scs_init()、lockdep_init_task()和uprobes_init()，分别用于初始化影子调用栈（Shadow
Call Stack），init进程（ init_task
）的锁依赖跟踪信息和用户态探针（uprobes）。影子调用栈用于增强安全性，防止返回地址攻击，锁依赖跟踪信息用于检测死锁，用户态探针用于调试和动态追踪。

简单来说，该函数的作用就是初始化进程创建所需的所有核心数据结构缓存并设置系统可创建进程数量上限。
