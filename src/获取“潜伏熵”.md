## 获取“潜伏熵”

在内核启动的最早阶段（磁盘还没转，网络还没通，甚至中断还没完全开启），随机数池是非常“饥饿”的。add_latent_entropy()
通过收集内核初始化过程中的执行差异来获取能量，从几乎没有任何外部噪声的环境中捕获微小的不可预测性。add_latent_entropy()
定义位于git/drivers/char/random.c，其定义为：

```
#if defined(LATENT_ENTROPY_PLUGIN) && !defined(__CHECKER__)
static inline void add_latent_entropy(void)
{
	add_device_randomness((const void *)&latent_entropy,  sizeof(latent_entropy));
}
#else
static inline void add_latent_entropy(void) {}
#endif
```

从上述定义可以看出，该函数必须配合编译开关LATENT_ENTROPY_PLUGIN使用。如果没有启用LATENT_ENTROPY_PLUGIN，函数为空函数，不起任何作用。

这个函数通常配合 GCC 的插件 (Latent Entropy Plugin) 使用。当内核开启了
CONFIG_GCC_PLUGIN_LATENT_ENTROPY 时，GCC
编译器会在某些内核函数的控制流中插入代码。每当内核执行这些特定的初始化代码时，变量会根据程序执行的路径（比如分支预测、缓存命中等微小的时间差异）产生累积的变化。add_latent_entropy()
会获取这个由“执行路径轨迹”产生的随机值，并将其混入熵池。

它主要是为了解决
无盘工作站或嵌入式设备的安全问题。这类设备在启动瞬间的状态非常固定（每次上电执行的代码指令序列几乎一模一样）。如果没有
add_latent_entropy
这种从软件执行逻辑中榨取随机性的机制，这些设备在生成第一个 RSA 密钥或
UUID 时可能会出现全球重复。

如果说 crng_initialize_primary 是在找“专门的随机源”，那么
add_latent_entropy 就是在从枯燥的初始化代码执行过程中“捡漏”，把由于 CPU
乱序执行或缓存抖动产生的微小差异转化为系统最初的安全防线。
