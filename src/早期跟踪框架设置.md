## 早期跟踪框架设置

early_trace_init() 是 Linux
内核启动过程中的一个关键初始化函数，负责在内核还没有完成完整 trace
子系统初始化之前，就先搭建一个“简化版”的 trace机制，用来记录早期启动过程中的调试信息。函数定义在git/kernel/trace/trace.c，其定义为：

```
void __init early_trace_init(void)
{
	if (tracepoint_printk) {
		tracepoint_print_iter =  kmalloc(sizeof(*tracepoint_print_iter), GFP_KERNEL);
		if (MEM_FAIL(!tracepoint_print_iter,  "Failed to allocate trace iterator\n"))
			tracepoint_printk = 0;
		else
			static_key_enable(&tracepoint_printk_key.key);
	}
	tracer_alloc_buffers();
}
```

其主要功能包括：

- 如果开启了tracepoint_printk，则利用kmalloc()函数从kmem分配一些内存，用以记录打印日志的状态（tracepoint
  print iterator），同时利用跳转标号修改技术启动早期日志打印。tracepoint_printk是一个通过内核启动参数tp_printk开启的功能。它允许将所有跟踪点（Tracepoints）输出的信息直接重定向到内核日志（printk/ dmesg）中

- 利用函数tracer_alloc_buffers()为日志分配环形缓冲区，初始化per-cpu缓冲区，建立trace存储空间

有关日志缓冲区内容，可参考第14章。

tracer_alloc_buffers()不仅分配内存，还构建了内核跟踪的最小运行环境。其主要工作包括：

- 首先通过 security_locked_down(LOCKDOWN_TRACEFS)确认系统是否处于“锁定”模式。如果内核开启了安全加固，出于防止敏感信息泄露的考虑，会禁止跟踪功能并返回权限错误

- 通过 alloc_cpumask_var() 分配 CPU 掩码，以确定哪些CPU核芯参与跟踪。检查内核中是否调用 trace_printk()，如调用则初始化专用缓冲区，以便在任何上下文中都能高效打印调试信息

- 如果缓存区曾经扩容且扩容成功，则使用trace_buf_size用设定ring_buf_size，否则，将ring_buf_size 设置为最小值

- 调用 cpuhp_setup_state_multi ()注册 CPU 热插拔回调。这样，当系统有新的
  CPU核芯上线时，内核会自动为该核心分配对应的环形缓冲区，确保跟踪在多核环境下的连续性

- 分配一个环形临时缓冲区（大小为一个页面），用于事件触发等特殊用途，同时调用
  allocate_trace_buffers() 正式为全局跟踪实例 global_trace 分配环形缓冲区

- 如果设置了trace_boot_clock，则通过 tracing_set_clock() 设置跟踪时间

- 设置global_trace结构体各个字段的值，以调用 register_tracer()注册跟踪器

- 调用init_function_trace()初始化函数跟踪

- 向内核注册 panic_notifier_list ()和 die_notifier()回调函数。这样当内核发生崩溃（Panic）或致命错误时，跟踪系统可以触发特定的回调逻辑（如保存当前的跟踪快照）

- 初始化 global_trace 结构体的各种链表头，加入到全局跟踪数组列表ftrace_trace_arrays 中

- 通过函数apply_trace_boot_options()使引导跟踪选项发生作用

- 注册快照触发器，完成初始化。
