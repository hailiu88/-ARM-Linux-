## 满足框架的中断特殊需求

中断初始化函数分两步完成，第一步由early_irq_init()完成，主要为满足硬件架构的特殊需求而设。该函数为一个弱函数，默认操作为返回
0 的空壳函数。其实现定义在git/kernel/softirq.c文件里。定义为：

```
int __init __weak early_irq_init(void)
{
	return 0;
}
```
在某些架构中，可能需要在分配任何中断之前预留特定的内存或设置特殊的硬件寄存器，此时，架构代码会重写这个函数。由于默认函数实现为弱函数，当系统出现一个同名函数时，系统会优先选取针对特定架构的同名函数。即使某个架构没有特殊的中断初始化需求，它也可以直接使用这个弱实现的默认行为，而不会导致链接失败。这种设计体现了
Linux 内核的 “通用框架 + 架构差异” 的解耦思想。

如果开启了稀疏中断（CONFIG_SPARSE_IRQ），它会重新实现erarly_irq_init()函数，其实现定义在文件git/kernel/irq/irqdesc.c文件里。函数的实现为：

```
int __init early_irq_init(void)
{
	int i, initcnt, node = first_online_node;
	struct irq_desc *desc;
	init_irq_default_affinity();
	initcnt = arch_probe_nr_irqs();
	printk(KERN_INFO "NR_IRQS: %d, nr_irqs: %d, preallocated irqs: %d\n",
	       	NR_IRQS, nr_irqs, initcnt);
	if (WARN_ON(nr_irqs > IRQ_BITMAP_BITS))
		nr_irqs = IRQ_BITMAP_BITS;
	if (WARN_ON(initcnt > IRQ_BITMAP_BITS))
		initcnt = IRQ_BITMAP_BITS;
	if (initcnt > nr_irqs)
		nr_irqs = initcnt;
	for (i = 0; i < initcnt; i++) {
		desc = alloc_desc(i, node, 0, NULL, NULL);
		set_bit(i, allocated_irqs);
		irq_insert_desc(i, desc);
	}
	return arch_early_irq_init();
}
```

在 Linux 内核中断子系统中，此版本的 early_irq_init
是稀疏中断描述符（Sparse IRQ
Descriptors）的核心初始化逻辑。其核心任务是构建中断管理的数据结构基础，并完成从静态编译配置到动态运行环境的过渡。

该函数通过init_irq_default_affinity()初始化中断的默认 CPU
亲和性掩码。在多核（SMP）系统中，它决定了在没有专门设置时，中断默认由哪些
CPU 核心响应，通常初始化为全集（即所有在线 CPU）。

它还通过函数arch_probe_nr_irqs()获取当前硬件平台预分配的中断向量数量，并通过
IRQ_BITMAP_BITS（通常由 NR_IRQS 派生）对系统总中断数（nr_irqs）和
预分配中断数（initcnt）进行硬约束。这确保了用于追踪中断分配状态的位图（Bitmap）不会发生越界访问。

在开启 CONFIG_SPARSE_IRQ
后，系统不再维护静态的中断描述符结构体数组，而是采用基数树进行稀疏管理。通过函数alloc_desc()在指定的内存节点上为每一个中断分配存储中断描述符结构体的内存，并通过调用
set_bit
标记该中断号在全局位图allocated_irqs中的位置已被占用，防止后续动态申请时发生冲突。之后通过函数irq_insert_desc()将分配的
irq_desc 实例插入到全局基数树中。基数树提供了从中断号（Key）到
描述符指针（Value）
的高效查找算法。这在处理具有成千上万个中断源的大型系统时，比线性表更具性能优势且内存利用率更高。

函数在最后执行
arch_early_irq_init()。这是一个硬件架构相关的钩子函数（例如在 x86
），用于完成特定硬件层级的早期映射。arm版Linux使用默认的空函数。
