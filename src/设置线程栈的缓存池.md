## 设置线程栈的缓存池

为了提高创建进程（fork）的速度，内核不会每次都从通用的内存分配器中申请栈空间，而是维护一个缓存池。当一个进程退出时，它的内核栈不会立即释放回系统内存，而是放入这个缓存池中，供下一个新进程直接提取使用。通过专用的
Slab/Slub 缓存管理栈内存，可以有效减少系统内存碎片。

每个进程在内核态运行时，都需要一个独立的内核栈（通常为 8KB 或
16KB）。它不同于用户态的栈，内核栈存储了系统调用时的寄存器现场、内核函数嵌套调用的局部变量等关键数据。

在 kernel/fork.c 中，内核通过不同的方式分配栈内存。如果定义了
CONFIG_VMAP_STACK，内核使用 vmalloc 分配栈（通常是为了加上 Guard Page
防止溢出）。此时，它会使用一套专门的 cached_stacks 逻辑（按 CPU
缓存），而不是通用的 thread_stack_cache。如果 THREAD_SIZE \>=
PAGE_SIZE，内核通常直接使用伙伴系统分配整页内存。

如果一个栈连一个页面（通常 4KB）都占不满（常见于某些嵌入式架构或 32
位系统），直接分配一页会非常浪费内存。这时内核会调用
thread_stack_cache_init ()，通过 Slab 分配器（kmalloc
的底层）创建一个名为 "thread_stack"
的专用池。在分配内存时，确保分配出的内存块精确等于
THREAD_SIZE，从而避免内存碎片。

线程栈的缓存池更多地存在于32 位架构（如 ARM32, MIPS）。由于其栈可能只有
4KB 或更小，通过 Slab 细粒度管理内存，可以改善内存的利用率。在 x86_64
等主流 64 位架构上，由于默认开启了 CONFIG_VMAP_STACK（栈通常为
16KB），不使用线程栈的缓存池。

thread_stack_cache_init()位于git/kernel/fork.c，其定义为：

```
void thread_stack_cache_init(void)
{
	thread_stack_cache = kmem_cache_create_usercopy("thread_stack",
				THREAD_SIZE, THREAD_SIZE, 0, 0, THREAD_SIZE, NULL);
	BUG_ON(thread_stack_cache == NULL);
}
```

thread_stack_cache_cache()
的作用非常简单：从内核缓冲区为内核线程栈的快速分配池分配一块名为thread_stack、大小为THREAD_SIZE的内存，用于处理关闭
VMAP_STACK 且线程所用栈小于一页的情况。
