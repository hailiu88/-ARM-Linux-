## 调试模式切换

从启动到运行，Linux
的调试模式分为早期调试模式和完整调试模式两个阶段，其核心区别在于系统资源的可用性。

早期调试模式的标志为dbg_is_early =
true。此时内核处于极早期，内存管理尚未完整建立，中断未开启，文件系统和设备驱动均不可用，通信只能依靠最原始的硬件接口（如
早期串口 earlycon 或 earlyprintk）。早期调试的核心任务为调试内核在
setup_arch() 阶段的崩溃，捕获在 LSM
或内存管理系统初始化之前的严重错误。在这一阶段，调试功能受限，通常只能进行简单的内存查看或单步执行。

完整调试模式的标志为dbg_is_early = false。此时内存管理、中断、多核调度
(SMP) 已完全就绪，可以使用完整的驱动程序（如标准的 ttyS0 串口、以太网
KGDBoE 等）。这时可以利用 KDB
交互式控制台进行复杂的内核状态分析，支持软断点和更复杂的架构相关调试特性。在完整调试模式下，全功能开启。如果在启动参数中设置了
kgdbwait，系统会在此阶段触发
kgdb_initial_breakpoint()，强制停下等待开发者接入。

dbg_late_init()
函数是从早期调试模式向完整调试模式转换的关键转折点，它标志着调试器从救急模式转向常规开发模式。dbg_late_init()函数的主要功能是状态切换与全面激活。它位于git/kernel/debug/debug_core.c，定义为：

```
void __init dbg_late_init(void)
{
	dbg_is_early = false;
	if (kgdb_io_module_registered)
		kgdb_arch_late();
	kdb_init(KDB_INIT_FULL);
	if (kgdb_io_module_registered && kgdb_break_asap)
		kgdb_initial_breakpoint();
}
```

函数首先通过赋值语句dbg_is_early =
false进行状态位切换，告诉系统：内存管理、中断系统等基本就绪，内核的早期启动阶段已经结束，调试器不再处于受限的早期模式。如果系统注册了KGDB
的 I/O
通道（调试通信接口），则调用kgdb_arch_late()，执行与处理器架构相关的后期初始化，准备与架构相关的调试功能。只有x86架构需要kgdb_arch_late()函数，对arm32或arm64，函数kgdb_arch_late()为弱的空函数。

kdb_init()负责初始化内核调试器（Kernel Debugger-KDB），正式进入
KDB的完全初始化状态。与 KGDB 不同，KDB
是一个内置在内核中的交互式命令行调试器，允许工程师直接在控制台输入命令（如
lsmod, ps, rd
查看寄存器等）。kdb_init()主要完成命令集注册、环境变量初始化及符号表链接，把所有的
KDB 内置命令到内存，设置KDB 的内部变量，将内核符号表关联到
KDB，使得开发者可以输入函数名。

在函数的最后，如果有通信通道且通过kgdb_break_asap（对应启动参数
kgdbwait）明确要求尽快断开，系统会利用函数kgdb_initial_breakpoint()设置一个软中断，停下来等待调试器GDB连接。
