## 搭建时钟节拍和广播框架

时钟节拍（Tick）是 Linux
内核的心跳。它本质上是一个由硬件定时器定期触发的周期性中断。如果没有这个时钟节拍，内核就像一块没有秒针的时钟，无法进行进程调度，也无法处理超时任务。

每当一个时钟节拍中断发生时，内核会执行以下操作：

- 增加 jiffies 计数器（系统开机以来的时钟节拍总数）

- 检查当前进程的运行时间片是否用完。如果用完，则触发调度器（CFS）进行切换

- 检查是否有低精度定时器（Timer Wheel）到期

- 更新 CPU 的用户态/内核态运行时间统计

现代 Linux 内核支持周期性时钟节拍 (Periodic Tick)和动态时钟节拍 (NO_HZ /
Tickless)两种时钟节拍模式。其中周期性时钟节拍由硬件定时器以固定频率（如
100Hz, 250Hz, 1000Hz）持续产生，即使 CPU 在休眠，也必须被不断唤醒
来处理这个中断，极其耗电。
动态时钟节拍有空闲（idle）和全满（full）两种工作方式。在时钟节拍机制工作于空闲方式时，当
CPU
进入空闲，内核会计算下一个定时器到期的时间，并把硬件闹钟设在那个点，中间这段时间停止时钟节拍，让
CPU
深度睡眠。在高性能计算或实时系统中，为了提高系统性能及响应时间，如果一个核只有一个任务，内核会完全关闭时钟节拍以减少中断干扰，这时时钟节拍机制工作于全满方式。

时钟节拍机制由函数tick_init()创建。函数在文件git/kernel/time/tick-common.c中实现，其定义为：

```
void __init tick_init(void)
{
	tick_broadcast_init();
	tick_nohz_init();
}
```

这段代码虽然只有两行，但它构成了 Linux
现代时间管理机制的顶层框架。它决定了系统是像老式闹钟一样“滴答滴答”不停地响，还是像智能手机一样只在需要时“醒来”。

tick_broadcast_init()为时钟节拍广播使用的一些掩码（位图）分配内存并将分配的内存清零，掩码包括tick_broadcast_mask、tick_broadcast_on、<u>tmpmask</u>等三个掩码，其中tick_broadcast_mask标识需要接收唤醒时钟节拍的CPU。如果开启了CONFIG_TICK_ONESHOT，还额外分配三个用于单纯节拍模式下的掩码，其中tick_broadcast_oneshot_mask用于记录需要由广播定时器代为触发节拍的
CPU。

时钟节拍广播是为了解决多核时代的“失聪”难题。在多核 (SMP) 系统中，每个
CPU 通常都有自己的本地定时器。当 CPU
进入深度睡眠时，它的本地定时器往往会停止工作。这时，如果没有外部干扰，这个
CPU 就会一直休眠，错过预定的任务。

为了解决这一问题，Linux准备了一个全局定时器（不随 CPU
睡眠而停止的硬件）。当某个内核要睡着时，它会将自己的唤醒任务托付给这个全局定时器。时间一到，全局定时器产生中断，由一个醒着的
CPU（通常是 CPU0）通过 IPI（处理器间中断）把睡着的 CPU
唤醒。tick_broadcast_init 初始化了这套广播机制。

传统的Linux采用周期性的时钟节拍，即无论有无任务可执行，每秒之内CPU都会被中断多次（中断次数用HZ表示，为一常量，在编译时设定，通常为100次或1000次）。如果CPU长期空闲，这样不停中断会非常耗能。为了解决这一问题，Linux设置了无时钟节拍模式或动态时钟模式。通过编译配开关CONFIG_HZ_PERIODIC（传统模式）、CONFIG_NO_HZ_IDLE（动态时钟空闲模式）和CONFIG_NO_HZ_FULL（全动态时钟模式）用户可以在周期性模式（传统模式）、动态时钟空闲模式和全动态时钟模式三种工作模式之间选择。在周期模式下，时钟节拍永不停歇。在动态时钟空闲模式下，当
CPU 进入 Idle（空闲）
状态时，停止周期性时钟中断。在全动态模式下，不仅在空闲时，甚至当 CPU
上只有一个任务在运行时，也会尝试关闭或减缓时钟。全动态时钟节拍模式通常用于实时系统。

tick_nohz_init()初始化无时钟节拍模式，这是 Linux
迈向低功耗和高性能的最重要步骤。该函数根据内核配置（如
CONFIG_NO_HZ_COMMON）和启动参数（如 nohz=on）初始化内部状态机和
static_key
标志位。函数tick_nohz_init()位于git/kernel/time/tick_sched.c文件中，其定义为：

```
void __init tick_nohz_init(void)
{
	int cpu, ret;

	if (!tick_nohz_full_running)
		return;
	if (!arch_irq_work_has_interrupt()) {
		pr_warn("NO_HZ: Can't run full dynticks because arch doesn't support irq work self-IPIs\n");
		cpumask_clear(tick_nohz_full_mask);
		tick_nohz_full_running = false;
		return;
	}
	if (IS_ENABLED(CONFIG_PM_SLEEP_SMP) &&
			!IS_ENABLED(CONFIG_PM_SLEEP_SMP_NONZERO_CPU)) {
		cpu = smp_processor_id();

		if (cpumask_test_cpu(cpu, tick_nohz_full_mask)) {
			pr_warn("NO_HZ: Clearing %d from nohz_full range ""for timekeeping\n", cpu);
			cpumask_clear_cpu(cpu, tick_nohz_full_mask);
		}
	}
	for_each_cpu(cpu, tick_nohz_full_mask)
		context_tracking_cpu_set(cpu);
	ret = cpuhp_setup_state_nocalls(CPUHP_AP_ONLINE_DYN,
					"kernel/nohz:predown", NULL,tick_nohz_cpu_down);
	WARN_ON(ret < 0);
	pr_info("NO_HZ: Full dynticks CPUs: %*pbl.\n", cpumask_pr_args(tick_nohz_full_mask));
}
```

该函数首先依据cpu是否正运行在无时钟节拍模式以及当前处理器架构是否支持通过硬件中断触发
irq_work，确定是否需要初始化无时钟节拍机制。如果系统是多核的，且开启了休眠功能，但该系统不允许或不支持在休眠过程中通过
CPU 热插拔来下线非启动内核，则打印警告信息。

接下来的代码利用记录了哪些 CPU
允许关闭周期性时钟中断的tick_nohz_full_mask
掩码（位图），遍历所有在系统启动参数（如
nohz_full=1-3）中被指定为全动态时钟模式的 CPU核芯，注册上下文跟踪和配置
CPU 热插拔回调函数。为了在关闭时钟节拍的情况下依然能准确记录 CPU
的执行上下文（user / kernel / idle），以支持 RCU 和 tickless
运行，必须开启上下文跟踪。上下文跟踪是 NO_HZ_FULL 正常运行的先决条件。

函数cpuhp_setup_state_nocalls()为CPU在热插拔框架中申请一个动态状态
ID，CPU热插拔前的状态为CPUHP_AP_ONLINE_DYN，命名该状态为"kernel/nohz:predown"，注册CPU在线时的回调函数为空，下线时的回调函数为tick_nohz_cpu_down。这里只注册了
teardown 回调函数 tick_nohz_cpu_down，用于在 CPU
即将下线时执行清理工作。

函数名最后部分_nocalls 表示在注册这一刻，不对当前已经在线的 CPU
立即调用这些回调函数，仅供未来发生热插拔事件时使用。

总的来说，最后一段for循环模块的功能是：在系统初始化阶段，标记特定的 CPU
进入全动态时钟模式，并确保当这些 CPU
未来因热插拔下线时，系统能正确地关闭其独有的定时器逻辑。

真正实现时钟节拍的是时钟事件设备 （Clock Event
Device）。在初始化之后，内核会寻找精度最高的设备（如 ARM 的 Generic
Timer 或 x86 的 LAPIC Timer）作为时钟节拍的物理载体。之后，内核会通过
set_next_event() 函数告诉硬件：“请在 xx纳秒后给我一个中断”。

时钟节拍在内核中的工作流程为：

- 底层的硬件定时器触发

- 执行中断处理函数，如 tick_handle_periodic

- 调用 do_timer() 更新全局 jiffies

- 调用 update_process_times() 并触发 TIMER_SOFTIRQ

> 这正是为什么 softirq_init() 必须在时间子系统完全运转前准备就绪的原因。
