## 热插拔内存管理初始化

多数内存器件的热插拔功能不依赖于CPU，但有些NUMA内存器件需要CPU的支持才能够实现热插拔功能。为此，Linux需要实现NUMA内存的热插拔管理功能，以便支持不能够单独实现热插拔功能的NUMA内存器件。

Linux把CPU的热插拔管理过程分为多个阶段或状态，大致可分为断线、静默、准备、上线等几个阶段。每个阶段又分为多个步骤，具体划分可参考文件git/include/linux/cpuhotplug.h文件中的
cpuhp_state枚举类的定义。Linux利用前面介绍过的cpuhp_cpu_state结构体
记录cpu在不同阶段的状态，主要包括CPU的当前状态， 目标状态，
正在运行的热插拔线程，处于启动（bringup）还是卸载（teardown）状态等多个域。

在热插拔的管理过程中，处于任何状态的CPU都有可能进入休眠、也有可能被从不同的状态唤醒。CPU在任何状态都有可能出错，也有可能从不同的状态恢复，因此，各个状态都要有相应的启动（startup）和卸载函数。为了保存不同状态下以回调函数方式定义的启动和卸载函数，Linux定义了类型为cpuhp_step结构体的数组cpuhp_hp_states，用以保存各个状态使用的回调函数。该数组定义在文件git/kernel/cpu.c中。

结构体cpuhp_step的定义为：

```
struct cpuhp_step {
	const char		*name;
	union {
		int		(*single)(unsigned int cpu);
		int		(*multi)(unsigned int cpu, struct hlist_node *node);
	} startup;
	union {
		int		(*single)(unsigned int cpu);
		int		(*multi)(unsigned int cpu, struct hlist_node *node);
	} teardown;
	struct hlist_head	list;
	bool			cant_stop;
	bool			multi_instance;
}
```

定义在git/kernel/cpu.c文件中，其中list域为一个哈希链表，保存调用本结构体中startup()和teardown()的所有实体（instance）。

在初始化numa节点的区域结构体后，start_kernel()函数调用page_alloc_init()，通过函数cpuhp_setup_state_nocalls()把热插拔的CPUHP_PAGE_ALLOC_DEAD状态中步骤mm/page_alloc:dead的卸载函数设置为page_alloc_cpu_dead()，即把在cpuhp_hp_states\[CPUHP_PAGE_ALLOC_DEAD\]处的结构体cpuhp_step中的name域置为mm/page_alloc:dead，把teardown.single域置为函数page_alloc_cpu_dead()的地址，把multi_instance域置为0，其余各个域的值为NULL。

函数page_alloc_cpu_dead()是内存管理模块中的内存释放函数，其主要任务是释放记录在当前CPU各种lru列表中的内存页面，把释放的内存页面记录到相应区域的zone结构体中的free_area数组，并更新释放内存对应的pglist_data结构体中的\_\_lruvec字段。\_\_lruvec类型为struct
lruvec，记录内存的最近使用状况（Least Recent Usage--LRU）。

为了高效释放内存，Linux依据内存使用状况（如内存是映射为磁盘文件还是映射为交换区域）把内存记录在不同的LRU链表中。struct
lruvec是各种类型lru链表的组合，有兴趣的读者可以参考Linux内存管理文献。

在进行内存管理时，
Linux使用page结构体描述内存页，用处不同，程序所处的运行状态不同，内存的特点及描述内存所用的变量就不同，因此page结构体定义也不同。Linux通过利用多个union数据类型定义page结构体，以方便page运用于不同的使用情况。page结构体定义在文件git/include/linux/mm_types.h文件里。

所有的page结构体的第一个域均为flags域，其中的每一位为一个标志位，用以表示不同类型或处于不同阶段的内存页面。这些标志位主要用来表示页面是否为复合页，页面是否为slab、页面是否活跃、内存页面是映射到磁盘文件还是交换区、页面内容是否可移出内存等。Linux所支持的页面类型定义在git/include/linux/page-flags.h文件中。

通常需要利用操作函数对这些标志位进行测试、置位、清除等操作。在使用查找工具查找这些标志位的操作函数时会发现找不到这些函数的定义，这是因为这些函数的定义是通过宏定义实现的，如宏定义PAGEFLAG(Unevictable,
unevictable, PF_HEAD)定义了对标志位PG\_
unevictable进行操作的函数。PAGEFLAG宏的定义为：

```
#define PAGEFLAG(uname, lname, policy)					\
	TESTPAGEFLAG(uname, lname, policy)					\
	SETPAGEFLAG(uname, lname, policy)					\
	CLEARPAGEFLAG(uname, lname, policy)
TESTPAGEFLAG、SETPAGEFLAG和CLEARPAGEFLAG的宏定义分别为：
#define TESTSETFLAG(uname, lname, policy)						\
static __always_inline int TestSetPage##uname(struct page *page)			\
	{ return test_and_set_bit(PG_##lname, &policy(page, 1)->flags); }
#define SETPAGEFLAG(uname, lname, policy)						\
static __always_inline void SetPage##uname(struct page *page)				\
	{ set_bit(PG_##lname, &policy(page, 1)->flags); }
```

和

```
#define CLEARPAGEFLAG(uname, lname, policy)						\
static __always_inline void ClearPage##uname(struct page *page)			\
	{ clear_bit(PG_##lname, &policy(page, 1)->flags); }
```

由此我们可以得出，PAGEFLAG(Unevictable, unevictable,
PF_HEAD)定义了TestSetPageUnevictable()，SetPageUnevictable()和ClearPageUnevictable()等三个对标志位PG\_
unevictable进行测试、置位和清零的函数。这些函数分别为：

```
static __always_inline int TestSetPageUnevictable(struct page *page)
{
	test_and_set_bit(PG_ unevictable, &PF_HEAD(page, 1)->flags);
}
static __always_inline void SetPageUnevictable(struct page *page)		
{ 
 set_bit(PG_ unevictable, & PF_HEAD(page, 1)->flags);
}
static __always_inline void ClearPageUnevictable(struct page *page)		
{ 
 clear_bit(PG_ unevictable, &PF_HEAD(page, 1)->flags);
}
```
