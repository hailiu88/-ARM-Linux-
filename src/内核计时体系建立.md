## 内核计时体系建立

在 Linux 内核中，计时系统（Timekeeping System）
是一套极其复杂的层次结构。它不仅要处理“现在几点”这种现实世界的问题，还要处理“运行了多久”以及“1微秒后唤醒我”这种高精度调度问题。

我们可以把内核计时系统拆解为 核心组件、时间类型 和 更新机制 三部分。

内核计时系统依赖时钟源、时钟事件设备和时间维护逻辑的相互配合
。时钟源通常是硬件计数器（如 CPU 内部的 TSC、主板上的 HPET 或 ACPI
Timer），它只负责不停地累加数值，内核通过它获取最精细的纳秒级增量。时钟事件设备是能够产生中断的硬件（如
LAPIC
Timer），内核用它来设定未来的某个时刻触发中断，从而驱动调度和定时器。时间记录器是内核的一个全局结构体，负责维护
CLOCK_REALTIME
等各种时间轴。它记录从时钟源读到的、并已转换为人类可理解的单位的时间。

内核不仅仅维护一个时间，它维护了多个并行的时间，主要包括：

- 墙上时间（CLOCK_REALTIME）

> 现实世界的 UTC 时间。可以被跳变（比如 NTP 校准或管理员手动修改）。

- 单调时间（CLOCK_MONOTONIC）

> 从系统启动开始计时。绝对不会回退，也不会受 NTP
> 突变影响，常用于计算任务耗时。

- 原始单调时间（CLOCK_MONOTONIC_RAW）

> 更原始的单调时间，完全不受 NTP 频率微调的影响。

- 引导时间CLOCK_BOOTTIME

> 包含系统休眠时间在内的单调时间。

- 原子钟时间（CLOCK_TAI）

> 原子时（考虑了闰秒偏移）。

计时系统的运转依赖于一个循环反馈机制。首先硬件触发，时钟事件产生周期性或随机性的中断。时钟事件处理函数tick_handle_periodic()调用tick_periodic()函数。tick_periodic()通过函数do_timer()更新全局
jiffies 和墙上时间，调用update_wall_time()校准时间维护器，然后通过调用
update_process_times()处理进程相关的本地事务。update_process_times()
调用 run_local_timers()，run_local_timers() 调用
hrtimer_run_queues()（高精度定时器）。如果发现低精度定时器中有任务到期，run_local_timers()会执行
raise_softirq(TIMER_SOFTIRQ)，告诉CPU有任务到期，需要处理。

时钟维护逻辑读取时钟源的当前计数值，根据预设的倍率（Mult）和位移（Shift）将其转换为纳秒，累加到时钟记录器（timekeeper结构体）的全局时间轴上。时钟记录器负责管理和分发系统时间的，连接了底层的硬件计数器和上层的时间
API。它是一个全局的单例结构体（通常由 tk_core
维护），负责维护内核中所有与时间相关的偏移量和状态。 当应用程序调用
gettimeofday() 或内核调用 ktime_get()
时，实际上是在读取时钟记录器维护的数据副本。

简单来说，时钟源提供了原始的数字脉冲，而时钟维护逻辑则负责把这些脉冲翻译成准确的时间记录在时钟记录器中，并确保整个内核和所有程序看到的表都是准的。

在内核启动序列中，timekeeping_init() 紧随 init_timers()
之后，是内核计时体系的核心初始化函数。函数定义在文件git/kernel/timekeeping.c里，其定义为：

```
void __init timekeeping_init(void)
{
	struct timespec64 wall_time, boot_offset, wall_to_mono;
	struct timekeeper *tk = &tk_core.timekeeper;
	struct clocksource *clock;
	unsigned long flags;
	read_persistent_wall_and_boot_offset(&wall_time, &boot_offset);
	if (timespec64_valid_settod(&wall_time) &&timespec64_to_ns(&wall_time) > 0) {
		persistent_clock_exists = true;
	} else if (timespec64_to_ns(&wall_time) != 0) {
		pr_warn("Persistent clock returned invalid value");
		wall_time = (struct timespec64){0};
	}
	if (timespec64_compare(&wall_time, &boot_offset) < 0)
		boot_offset = (struct timespec64){0};
	wall_to_mono = timespec64_sub(boot_offset, wall_time);
	raw_spin_lock_irqsave(&timekeeper_lock, flags);
	write_seqcount_begin(&tk_core.seq);
	ntp_init();
	clock = clocksource_default_clock();
	if (clock->enable)
		clock->enable(clock);
	tk_setup_internals(tk, clock);
	tk_set_xtime(tk, &wall_time);
	tk->raw_sec = 0;
	tk_set_wall_to_mono(tk, wall_to_mono);
	timekeeping_update(tk, TK_MIRROR | TK_CLOCK_WAS_SET);
	write_seqcount_end(&tk_core.seq);
	raw_spin_unlock_irqrestore(&timekeeper_lock, flags);
}
```

如果说 init_timers() 是在建立“闹钟管理框架”，那么 timekeeping_init()
就是在“校准系统大钟”。

init_timers() 处理的是相对时间，而 timekeeping_init()
处理的是绝对时间。它初始化系统时间（wall time）、启动时间（boot
time）、单调时间（monotonic time）以及时钟源。

墙上时间/Wall
Time以及系统运行时间/Uptime主要功能包括可大致划分为三个部分：

- 从主板上的实时时钟（RTC）芯片读取当前的年月日时分秒

- 初始化时间记录器 (timekeeper)

> 这是一个全局单例结构，存储了系统当前的各种时间戳（如
> CLOCK_REALTIME、CLOCK_MONOTONIC 等）。

- 关联时钟源

> 选择一个高精度的硬件计时器（如 TSC 或 HPET）作为后续计时的参考基准。

该函数用tk_core.timekeeper作为时钟记录器。它首先用read_persistent_wall_and_boot_offset()获取持久的墙上时间
(通常来自 RTC)
和启动偏移量，然后检查获取的时间是否有效。如果无效，则将其重置为 0
（即将墙上时间置为1970年-01-01）并发出警告，之后通过函数timespec64_compare()比较wall_time和boot_time的大小。如果wall_time小于boot_time，则将boot_time置为0。

Linux使用三种重要时间，分别为墙上时间（wall_time，从1970年开始算）、单调时间（monotonic
time，从系统开机开始算，不包含休眠时间）和启动时间（boot_time，从系统开机开始算，包含休眠时间），之间的关系为：

$$
montonic = wall\\_time + wall\\_to\\_mono
$$

其中wall_to_mono表示墙上时间到单调时间的偏移。

timespec64_sub(boot_offset,
wall_time)计算墙上时间到单调时间的偏移，这确保了上面的等式成立。

Linux支持多核、多线程运行。为了保证系统的一致性，对许多内存的访问均采用不同类型的锁。后面的代码在获取时钟记录器的自旋锁后，负责初始化时钟记录器结构体的各个字段，包括初始化网络时钟协议状态机，调用
tk_setup_internals 将默认时钟源与 timekeeper 结构体关联，设置 timekeeper
的初始墙上时间，调用 timekeeping_update
更新全局时间状态，并通知系统时钟已被设置（TK_CLOCK_WAS_SET）。初始化过程中使用顺序锁进行，以确保在此过程中读取时间的用户可以无锁访问。

timekeeping_init()
初始化后，系统才有了精确的纳秒级时间。之后每当硬件时钟产生中断（Tick），中断处理程序会通过
update_process_times() 调用软中断 TIMER_SOFTIRQ，并最终通过
timekeeping_update() 更新 timekeeper 中的时间，确保“大钟”不停走动。
