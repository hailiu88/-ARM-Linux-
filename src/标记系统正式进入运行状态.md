## 标记系统正式进入运行状态

通过将全局变量system_state标记为SYSTEM_RUNNING，标志系统已进入正式运行状态。之后，kernel_init()通过函数numa_default_policy()将当前进程（或内核线程）的内存策略重置为系统的初始默认状态。

在系统启动期间，为了缩短启动时间，内核通常会开启加速（Expedited）模式。这种模式下，RCU
宽限期会通过强制处理器中断等激进手段快速完成。

由于启动过程基本结束，kernel_init()通过函数rcu_end_inkernel_boot()调用停止这种激进模式。因为加速模式虽然快，但会消耗大量
CPU 资源并增加系统抖动，不适合系统正常运行时期。

函数rcu_end_inkernel_boot()位于git/kernel/rcu/update.c，定义为：

```
void rcu_end_inkernel_boot(void)
{
	rcu_unexpedite_gp();
	if (rcu_normal_after_boot)
		WRITE_ONCE(rcu_normal, 1);
	rcu_boot_ended = true;
}
```

该函数首先利用rcu_unexpedite_gp()取消之前的RCU加速调用，然后判断rcu_normal_after_boot是否为真，如果为真值，则强制将rcu_normal变量置为1。一旦
rcu_normal 为 1，所有的 RCU 同步请求（如
synchronize_rcu）都将使用标准的、低开销的路径，而不再尝试加速。

rcu_normal_after_boot是通过module_param()
宏定义的一个模块变量。module_param()的工作原理并非简单的变量赋值，而是一套复杂的编译时静态注册与运行时解析机制。它的核心任务是将一个
C 语言变量暴露给内核启动命令行（cmdline）或 insmod/modprobe 命令。

编译module_param(name, type, perm) 时，并不会生成可执行代码，而是利用
GCC 的属性（\_\_attribute\_\_((section(...)))）生成一个 struct
kernel_param 类型的描述符。这个描述符被放置在二进制文件的特定 ELF
段中（通常是 \_\_param 段）。它包含了变量的地址、类型对应的操作函数（如
param_ops_int）以及 sysfs 文件权限。 系统启动时，内核会扫描 \_\_param
段中的所有描述符。如果发现命令行（如 /proc/cmdline）中有匹配
模块名.参数名=值 的字符串（对应本函数的为update.
rcu_normal_after_boot=1的字符串），内核会调用对应的 set
回调函数将字符串解析并写入变量。也就是说，通过module_param()定义的变量可以通过内核启动命令行参数或模块调入参数改变其值。

rcu_boot_ended
是全局布尔标志位，作用是告知内核其他子系统（如调度器或内存管理器）系统的启动状态。一旦为真，标志启动阶段已正式结束

kernel_init()还要调用do_sysctl_args()处理启动命令行中类似sysctl.键=值的启动参数。函数位于git/fs/proc/proc_sysctl.c，定义为：

```
void do_sysctl_args(void)
{
	char *command_line;
	struct vfsmount *proc_mnt = NULL;
	command_line = kstrdup(saved_command_line, GFP_KERNEL);
	if (!command_line)
		panic("%s: Failed to allocate copy of command line\n", __func__);
	parse_args("Setting sysctl args", command_line, NULL, 0, -1, -1, &proc_mnt, process_sysctl_arg);
	if (proc_mnt)
		kern_unmount(proc_mnt);
	kfree(command_line);
}
```

该函数解析启动参数中以 sysctl. 开头的配置项，它最后两个参数是 &proc_mnt
和
process_sysctl_arg。由于第三个参数为NULL，在command_line中的所有参数都会被当作未知参数而由process_sysctl_arg
回调函数处理。回调函数位于git/fs/proc/proc_sysctl.c，其主要作用包括：

- 参数过滤与对齐

> 函数首先检查参数是否以 sysctl 开头，如果是
> sysctl.vm.swappiness，它会跳过 sysctl 这 6 个字符，如果不是以 sysctl
> 开头，会通过 sysctl_find_alias 查找（例如一些老旧的参数别名）。它支持
> . 或 / 作为分隔符（如 vm.swappiness 或 vm/swappiness）。

- 动态挂载 procfs

> 为了写 sysctl 设置，必须访问 /proc/sys/。它不会一直挂载着
> proc，只有在发现第一个 sysctl 参数时，才调用 kern_mount
> 进行内核内部挂载。

- 路径转换与文件操作

> 使用 kasprintf 拼出完整路径，例如将 vm.swappiness 变成
> sys/vm.swappiness。调用 strreplace(path, '.', '/')
> 将所有的点号换成斜杠，最终路径变为 sys/vm/swappiness（相对于 proc
> 根目录）。

- 模拟写入

> 使用 file_open_root 打开对应的伪文件，使用 kernel_write 将启动参数中的
> val 写入该文件。这就触发了该 sysctl 节点对应的
> proc_handler（内核函数），从而安全地修改了内核变量。

- 报错处理

> 代码对各种错误（文件不存在、权限不足、值非法）进行了详细的 pr_err
> 打印。这也就是为什么当你写错启动参数时，可以在 dmesg
> 中看到具体的失败原因。

这个函数本质上是内核级别的 sysctl -w
命令。它利用已有的虚拟文件系统接口来保证修改的安全性和一致性，而不需要为每一个内核参数编写专门的解析代码。

do_sysctl_args()解析完成后，释放临时备份并卸载临时挂载的 proc
文件系统。sysctl 的本质是操作 /proc/sys/
下的文件。在内核启动阶段，真正的 procfs 可能还没完全挂载到用户态。为了让
process_sysctl_arg
能够像操作文件一样设置内核参数，它可能会内部临时挂载一个 proc 实例。

这个函数是内核自动化配置的一部分。它确保在 GRUB 里写的例如
sysctl.vm.swappiness=10 这种参数，在内核启动的早期阶段就能被正确应用。

do_sysctl_args()函数运行结束后，准备工作已全部结束，接下来的工作是kernel_init()函数最核心的目标：启动用户空间init程序。
