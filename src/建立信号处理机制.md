## 建立信号处理机制

在 Linux
中，信号（Signal）是内核与进程之间、或进程与进程之间的一种异步通信机制。它类似于软件层面的中断。信号处理主要分为生成、挂队、传递与处理三个阶段。信号可以由硬件异常、终端操作、软件请求、内核事件等多种情况触发。

内核不会立即打断进程去处理信号，而是先在进程描述符（task_struct）的pending
字段中做一个标记，记录哪些信号到达了。pending
字段事实上是一个包含signal和list两个字段的sigpending结构体，其中signal是一个位图变量，每一位代表一类信号（1表示信号发生），list是信号队列队头。信号队列存在于进程描述符，它存储了信号的详细附加信息（如发送者信息，发送的原因等），而signal位图则是为了快速查找信号。如果进程设置了信号掩码（Signal
Mask），信号会一直处于未决（pending）状态，直到屏蔽解除。

内核在进程从内核态返回用户态的瞬间（例如系统调用结束或时钟中断结束时），会检查是否有未决信号。

信号处理方式有三种：

- 忽略（Ignore）

> 丢弃信号，什么都不做（但 SIGKILL 和 SIGSTOP 不能被忽略）。

- 默认动作（Default）

> 执行系统默认行为，通常是终止进程、崩溃并产生核心转储（Core
> Dump）或停止进程。

- 捕获（Catch）

> 执行用户自定义的信号处理函数（Signal Handler）。

当进程捕获信号时，进程因为中断或系统调用进入内核。内核发现有信号要处理，于是修改进程的用户栈，伪造一个返回地址并返回用户态，跳转到信号处理函数执行。处理函数执行完后，通过特殊的系统调用（sigreturn）再次进入内核，内核恢复之前的上下文，让进程回到最初被打断的位置继续运行。

函数signals_init()的主要功能是分配用于存储信号结构体对象的缓存区、检查信号体大小及位置。函数位于git/kernel/signal.c，定义为：

```
void __init signals_init(void)
{
	siginfo_buildtime_checks();
	sigqueue_cachep = KMEM_CACHE(sigqueue, SLAB_PANIC);
}
```

signals_init()首先在编译阶段进行结构体一致性静态检查，确保内核与用户态程序在解释信号内容时，所有的字节偏移量完全对齐。之后，该函数通过宏KMEM_CACHE()分配缓存区，存放待处理信号的结构体，如果不能分配，则直接让内核崩溃。每发送一个信号（比如
kill -9），内核就要用该缓存区的一个 sigqueue 来填入发送者
PID、信号值等信息。
