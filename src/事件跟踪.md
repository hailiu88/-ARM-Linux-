## 事件跟踪

在 Linux 内核中，Trace Event 是一套基于 探测点（Tracepoints）
的高性能观测框架。它允许在不重启内核的情况下，动态监控内核的运行状态，或在不触发系统显著延迟的情况下，从内核关键路径中提取结构化数据。

事件跟踪可以从静态插桩、运行时控制和数据存储三个维度进行理解。内核开发者不会手动编写复杂的跟踪代码，而是使用
TRACE_EVENT() 宏进行静态插桩。这个宏在编译时会生成四种代码：

- 定义数据在环形缓冲区（Ring Buffer）中的二进制布局。

- 生成一个名为 trace_xxx() 的内联函数，放置在内核源代码中。

- 生成 /sys/kernel/tracing/events/.../format 文件，告诉用户态工具（如
  perf 或 trace-cmd）如何解析二进制数据。

- 生成开启/关闭该事件的钩子。

Trace Event
基于标号跳转技术，实现了近乎零的关闭开销。在内核二进制代码中，调用处被替换为一条
nop 指令。CPU 执行到这里直接跳过，开销仅为 1
个时钟周期。当需要跟踪事件时，即向 enable 文件写入 1
时，通过标号跳转技术，把NOP指令修改为跳转到相应的事件跟踪程序trace_xxx()，实现运行时控制。

在切换跟踪函数指针时，内核使用 RCU 机制，确保所有 CPU
都已经跨过了旧的执行路径，防止在函数指针更换瞬间发生崩溃。

当事件触发时，数据不会直接打印到屏幕，而是在 per-CPU
的环形缓存区中申请一块空间，然后执行 TP_fast_assign
中的逻辑，将内核变量（如 pid,
address）拷贝到缓存区，进行环形缓存。用户态进程读取 /trace
文件时，内核才将二进制数据转换成人类可读的字符串。

事件跟踪子系统按层次结构方式进行组织，由底层、探测层、控制层和顶层组成。各层的名称和作用分别为：

> 层次 组件 作用
>
> 顶层 Tracefs 用户接口 (/sys/kernel/tracing)，用于开关和过滤。
>
> 控制层 Ftrace 核心引擎，管理所有的事件、函数跟踪和环形缓存区。
>
> 探测层 Tracepoint 具体的插桩点，负责在内核执行流中拦截信息。
>
> 底层 RCU / Jump Label 支撑架构，保证跟踪的高效与安全。

事件跟踪不仅仅是记录，它还支持内核态的逻辑判断，进行过滤与触发：

- 可以通过 echo "common_pid == 1234" \> filter
  只记录特定进程的事件，减少数据干扰。

- 可以设置触发动作，例如，“当这个事件触发 10 次后，开启另一个跟踪器” 或
  “触发时记录一次堆栈”。

跟踪事件将静态代码中的探测点与用户空间的控制接口联系起来。它通过插桩，在内核源码中使用
TRACE_EVENT()
宏定义事件。当事件未开启时，它仅表现为一小段被跳过的指令（通常是
nop），对性能几乎无影响。

用户可以直接通过
Tracefs与内核的事件跟踪子系统交互。无需编写任何代码就可以查看所有可用事件、开启特定事件、查看输出。

示例：

查看所有事件：

> ls /sys/kernel/debug/tracing/events/

开启事件跟踪（如调度器事件）：

> echo 1 \> /sys/kernel/debug/tracing/events/sched/sched_switch/enable

查看输出：

> cat /sys/kernel/debug/tracing/trace

一个典型的 Trace Event
由TP_PROTO、TP_ARGS、TP_STRUCT\_\_entry、以及TP_fast_assign等四部分组成，分别用于定义传递给跟踪点的参数、与协议匹配的参数名、定义二进制环形缓冲区中存储的数据结构、以及将内核参数赋值到缓冲区的高效逻辑。
