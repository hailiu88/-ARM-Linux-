## 序列文件初始化

序列文件（seq_file ）是 Linux
内核中一套极其通用的接口，专门解决内核向用户态输出大量数据的问题。在读取
/proc/meminfo、/proc/net/dev 或 /proc/interrupts
等文件时，内核底层几乎都在使用序列文件。

传统的 read() 系统调用在处理 /proc
文件时，内核每次只能给用户态返回一页左右的数据。如果内核数据有几百
KB，开发者就得处理复杂的断点记录和偏移量逻辑。如果在两次读取之间，内核的数据结构变了（比如某个进程退出了），用户读到的数据可能会错位。

序列文件把复杂的分页和偏移逻辑封装好了，内核开发者只需要实现几个简单的回调函数。在使用序列文件时，开发者需要填充一个seq_operations
结构体。该结构体包含start、show、next、stop四个函数指针字段。start定位到要读取的起始位置（通常是行号），并加上必要的锁（避免读取时数据被修改），show负责把一个内核对象（比如一个进程、一个网卡信息）格式化成字符串，写入
序列文件的内部缓冲区，next告诉内核如何找到下一个元素（比如链表的下一个节点），stop结束读取，释放
start 中持有的锁。

在用户态执行 cat /proc/my_file时，内核通过系统提供的标准函数
seq_read()，反复调用 start -\> show -\>
next，直到把内核预设的缓冲区（通常是一页）填满，然后将缓冲区数据一次性拷贝给用户。如果数据没传完，序列文件会记住当前的
pos，下次用户再读时从这里继续。

如果 show
输出的数据太大，超过了缓冲区，序列文件会自动尝试分配更大的缓冲区并重试。开发者不需要关心
copy_to_user 的细节，只需要用 seq_printf 像写 printf 一样写逻辑即可。

函数seq_file_init()的作用是通过宏KMEM_CACHE()为 seq_file 对象建立专用
slab 缓存。函数位于git/fs/seq_file.c，其定义为：

```
void __init seq_file_init(void)
{
	seq_file_cache = KMEM_CACHE(seq_file, SLAB_ACCOUNT|SLAB_PANIC);
}
```

这个函数的作用是内核在系统启动时预先开辟一个slab内存池，专门用来存放
struct seq_file 结构体。如果失败系统会直接崩溃。SLAB_ACCOUNT
标志意味着该部分内存开销会计入进程的控制组配额。用户进程可以打开成千上万个
/proc 文件。如果没有
SLAB_ACCOUNT，恶意进程可能会通过大量打开文件来耗尽系统内存。有了它，内核就能限制单个容器或用户组消耗的
seq_file 内存。
