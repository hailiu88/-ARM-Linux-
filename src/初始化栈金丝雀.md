## 初始化栈金丝雀

栈金丝雀机制是Linux对栈进行保护的一种机制。所谓栈金丝雀（Canary）就是在函数调用栈的局部变量和返回地址之间插入一个随机数值。如果发生栈溢出攻击，这个值会被覆盖。内核在函数返回前会检查该值，如果发现变了，立即触发
内核崩溃并 停止系统，防止黑客获取控制权。

函数boot_init_stack_canary()的主要任务是初始化栈金丝雀值，是内核安全加固的关键一步。它的实现与系统的框架结构密切相关，其实现大都位于git/arch/处理器/include/asm/stackprotector.h文件里，而在git/include/linux/stackprotector.h中只定义了一个空函数。对arm处理器而言，该函数位于git/arch/arm/include/asm/stackprotector.h文件里，其实现为：

```
static __always_inline void boot_init_stack_canary(void)
{
	unsigned long canary;
	get_random_bytes(&canary, sizeof(canary));
	canary ^= LINUX_VERSION_CODE;
	current->stack_canary = canary;
#ifndef CONFIG_STACKPROTECTOR_PER_TASK
	__stack_chk_guard = current->stack_canary;
#else
	current_thread_info()->stack_canary = current->stack_canary;
#endif
}
```

这段代码清晰地展示了内核如何为 0号进程（swapper/idle 进程）
设置第一道安全防线。它首先利用函数get_random_bytes()获取的随机数，与内核版本号常量（
LINUX_VERSION_CODE）异或后赋予金丝雀变量canary。这是一种“混淆”手段。即使随机数生成器在极早期的初始化中出现了某种偏差，叠加一个编译时确定的版本号也能增加
Canary 值的差异性。

函数将生成的金丝雀变量值存入当前进程（0号进程）的 task_struct
结构体中，具体位置为结构体的stack_canary字段。后续系统创建新进程（fork）时，子进程通常会继承或重新生成这个值。这里的current为宏定义，用于从当前进程的thread_info结构体获取当前进程的task任务结构体。该宏定义用处非常广泛，其定义为：

\#define current current_thread_info()-\>task

而函数current_thread_info()则是通过全局指针变量current_stack_pointer获取当前进程的thread_info结构体的地址。thread_info结构体用于保存进程的元数据。

这段代码通过宏定义处理了全局和局部两种不同的栈保护方式。如果没有启用
CONFIG_STACKPROTECTOR_PER_TASK ，将随机值赋给全局变量
\_\_stack_chk_guard。如果启用了局部保护，则将值赋予当前进程的stack_canary字段。对早期的
GCC 栈保护实现或某些特定的架构，所有进程共用同一个 Canary
值，而对于采用了现代安全加固的系统，如
x86_64或arm，每个进程/线程拥有独立的 Canary
值。这样即使黑客破解了一个进程的 Canary，也无法直接用于其他进程。

该函数完成了从“获取随机数”到“部署防护点”的闭环。到这一步之后，内核的函数调用开始受到硬件和软件的双重监控。
