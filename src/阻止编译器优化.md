## 阻止编译器优化

函数arch_call_rest_init()运行结束后，共有三个线程在运行。start_kernel()对应线程0，线程1是否已进入用户态取决于调度器，我们无从而知。线程0已进入空循环等待新的任务。不论PID
0还是PID1，函数arch_call_rest_init()后面的函数prevent_tail_call_optimization()都没有机会执行。该函数只是一个简单的宏mb()，其作用是阻止编译器优化，强制要求
start_kernel 的栈现场保持原样。

到此为止，内核引导已完成其所有工作，接下来脚本解释器的启动、图形界面系统的启动等已是用户态程序（systemd）的工作。

在 Linux 的世界里，systemd
不仅仅是一个初始化程序（init），它已经演变成了一个庞大的系统管理平台。它的核心目标是：在
PID 1
的位置上，以最快、最有效率的方式把系统运行起来，并接管后续的一切管理工作。

它的主要功能可以归纳为以下几个核心维度：

- 进程启动与管理 (init 职责)

> 这是它最基础的功能，取代了传统的 SysVinit。传统的 init
> 是一个接一个串行启动脚本，速度慢。systemd 通过套接字激活（Socket
> Activation）等技术，让服务能够并行启动，极大缩短开机时间。通过配置文件（Unit
> 文件）清晰定义谁必须在谁之后启动，处理复杂的逻辑关系。如果某个服务（如
> Nginx）崩溃了，systemd 可以自动把它运行起来。

- 核心架构：单元 (Unit)

> systemd 把系统中的一切资源都抽象为 Unit，常见的有：

```
 .service：守护进程（如 Web 服务器）。

 .target：一组服务的集合（类似于传统的运行级别，如 multi-user.target）。

 .mount / .automount：挂载点管理。

 .timer：取代 Cron 的定时任务。
```

<!-- -->

- 日志管理 (journald)

> systemd 引入了
> journald，它采用二进制格式，不再是纯文本文件。日志记录更快且带索引，方便检索。每条日志会自动附带产生者的
> PID、UID、SELinux 上下文等，查错极快。通过 journalctl
> 命令可以查看系统启动以来的所有信息。

- 动态资源管理

> systemd 深度利用内核的 cgroups 功能。可以非常简单地在 Unit
> 文件里限制某个进程只能用 512M 内存或 10% 的
> CPU。当有连接请求进入特定端口时，才瞬间启动Socket
> 对应的服务。当有程序调用某个 D-Bus 接口时，才启动D-Bus 的目标服务。

- 其他系统组件 (生态圈)

> systemd 现在还包含了许多原本独立的系统工具：

- systemd-logind用于管理用户登录、Session 和电源管理（如关机、休眠）。

- systemd-resolved用于管理网络名字解析（DNS）。

- systemd-timesyncd用于替代 NTP 客户端同步时间。

- systemd-networkd用于管理网络配置和虚拟网络设备。

Systemd 启动后会检查默认目标（Default Target）。如果设置的是
graphical.target，它就会启动显示管理器（Display Manager）。显示管理器
(如 GDM, SDDM, LightDM)是我们看到的登录界面。它会启动 X Server 或
Wayland
Compositor（图形服务的底层基座）。输入密码登录后，显示管理器会启动窗口管理器和桌面组件（Panel,
Dock 等）。
