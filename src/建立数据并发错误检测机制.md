## 建立数据并发错误检测机制

KCSAN (Kernel Concurrency Sanitizer) 是 Linux
内核中的一种动态并发错误检测工具，专门用于在运行时捕获代码中的数据竞争
(Data Races)。 自 Linux 5.8
版本引入以来，它已成为内核开发者排查多线程同步问题的利器。
当两个线程同时访问同一个内存地址，且至少有一个访问是写操作时，如果这些访问中至少有一个是普通的（非原子的）内存访问，就会发生数据竞争。

KCSAN
不会监控每一次内存访问，它采用了一种基于观察点的随机抽样机制。内核运行过程中，KCSAN
会随机挑选一些内存访问动作。针对被选中的地址，KCSAN
会设置一个微小的延迟（Stall），并在此地址上挂一个观察点。如果在延迟期间，另一个线程也访问了同一个地址且触发了观察点，KCSAN
就会立即记录一个潜在竞争。它还会在延迟前后比对该内存地址的值，如果值发生了意外变化，即使没抓到现行，也会推断出存在竞争。

在内核代码中，KCSAN
通过编译器在所有的内存访问（读/写）前后都插了一段小脚本。当 CPU A
准备访问地址 X 时，它在 KCSAN
的内部哈希表里登记正在访问X的信息，并存入当前值，即挂上观察点。CPU A
故意停下来（Stall）几微秒，什么都不干。此时，如果 CPU B 也要访问地址
X，因为编译器也给 CPU B 插了脚本，它在访问 X 之前，会先去查一下 KCSAN
的哈希表。如果CPU B 发现地址 X 已经被 CPU A 登记监控了，CPU B 就知道A
也正在操作的数据，存在数据竞争的风险，于是打印出两边的调用栈。

在 KCSAN
里，观察点是软件模拟的，它利用一段共享的内存区域（哈希表）来记录哪些地址正在被观察。触发观察点本质上就是在第二个
CPU 在执行内存访问前，通过查表发现第一个 CPU
正在对自己要访问的地址进行延时监控。

很多开发者容易混淆KCSAN
v和KASAN这两个工具，其实它们的目的完全不同。KCSAN的主要目标是捕获并发及多线程逻辑
Bug，而KASAN的主要目标是捕获 内存非法访问 (越界、释放后使用)。

要开启KCSCAN，首先要在内核编译配置中设置 CONFIG_KCSAN = y，然后通过
/sys/kernel/debug/kcsan 接口，实时开启/关闭检测，或者查看竞争统计信息。

函数kcsan_init()的主要功能搭建检测机制。它位于git/kernel/kcscan/core.c，定义为：

```
void __init kcsan_init(void)
{
	BUG_ON(!in_task());
	kcsan_debugfs_init();
	prandom_seed_full_state(&kcsan_rand_state);
	if (kcsan_early_enable) {
		pr_info("enabled early\n");
		WRITE_ONCE(kcsan_enabled, true);
	}
}
```

函数首先通过
in_task()确保自己没在错误的时间（比如中断里）启动，因为初始化涉及创建调试文件和分配内存。如果没有运行在任务上下文，直接让内核奔溃。

函数kcsan_debugfs_init()的作用是在 /sys/kernel/debug/kcsan
创建调试接口，即生成名为kcsan的虚拟文件，并注册定义在debugfs_ops中的一系列回调函数。这样，用户可以通过命令行（如
echo 1 \>
/sys/kernel/debug/kcsan/enable）在系统运行时手动控制检测器的开关。debugfs_ops类型为file_operations结构体，包含read、write、open、llseek等三十多个函数指针字段。

函数通过调用 prandom_seed_full_state遍历所有
CPU，为每个CPU生成独立的随机种子并初始化状态，然后对伪随机状态进行预扰动。这是
KCSAN
的灵魂，它不会监控每一次内存访问，而是通过随机数决定“抽查”哪一次。这样既能抓到
Bug，又不会让系统卡死。

执行完 kcsan_init
后，内核就具备了抓捕数据竞争的能力。真正的错误检测会在随后的系统运行中，由各个
CPU 在读写内存时随机触发。
