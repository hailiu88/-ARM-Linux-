基于ARM的Linux操作系统引导详解

建立安全子系统架构

Linux 的安全架构是一个多层级、模块化的深度防御体系，从硬件底层一直延伸到用户应用层。其核心设计理念是即使一层被攻破，后续层级依然能提供保护。

LSM (Linux Security Modules) 是 Linux 内核中的一个通用安全框架。它并不直接提供某种特定的安全政策，而是为内核提供了一套钩子，允许不同的安全模块插入到内核中，以实现不同的访问控制模型。

在传统的 Linux 安全中，使用的是自主访问控制 (DAC- Discretionary Access Control)，即文件的拥有者可以决定谁能访问文件。

LSM 的作用是在 DAC 检查之后再进行一次检查。即使是 root 用户或文件拥有者，如果安全模块（如 SELinux）的策略不允许，依然无法操作。这种机制被称为强制访问控制 (MAC- Mandatory Access Control)。

LSM 在内核处理关键对象（如文件、进程、网络套接字等）的路径上放置了许多钩子函数。用户进程尝试执行一个操作（如 open() 一个文件）时，内核首先进行标准的 DAC 检查（权限位 rwx）。如果 DAC 通过，内核会调用 LSM 框架。LSM 调用已加载的安全模块（如 AppArmor），模块根据预设的安全策略决定允许或拒绝。只有当所有检查都通过时，内核才会真正执行该操作。

目前 Linux 内核官方支持并常用的 LSM 模块包括：

  • SELinux (Security-Enhanced Linux)

功能最强大，基于标签控制，常用于 RHEL/CentOS 和 Android。

  • AppArmor

基于路径名的控制，配置相对简单，常用于 Ubuntu/Debian。

  • Smack

  • 简化的强制访问控制，最初为嵌入式设计。

  • Yama

  • 专门用于限制进程间的 ptrace 操作，防止敏感内存被读取。

  • Landlock

较新的非特权 LSM，允许应用程序沙盒化自身，将程序隔离在一个封闭、受限的虚拟环境（即“沙盒”）中运行，以保护操作系统和真实数据。

即使程序被黑客攻破，LSM 也能限制黑客只能在极小的范围内活动，无法获取系统核心权限。开发者无需修改内核核心代码，就能通过编写 LSM 模块来实现自定义的安全逻辑。作为系统的最后一道防线，LSM能够有效防御提权攻击和容器逃逸。

security_init()函数是 Linux 内核中 LSM（Linux Security Modules）安全模块框架的核心初始化函数。其主要作用是在系统启动早期，按照特定顺序加载并初始化各个安全模块（如 SELinux, AppArmor, Smack 等），函数位于git/security/security.c，定义为：

int __init security_init(void)
{
	struct lsm_info *lsm;
	pr_info("Security Framework initializing\n");
	for (lsm = __start_early_lsm_info; lsm < __end_early_lsm_info; lsm++) {
		if (lsm->enabled)
			lsm_append(lsm->name, &lsm_names);
	}
	ordered_lsm_init();
	return 0;
}

该函数首先通过 pr_info 在内核日志中打印日志，接着遍历从 __start_early_lsm_info 到 __end_early_lsm_info 地址段中的所有 lsm_info 结构体，收集早期 LSM 模块名称，将所有处于 启用状态的早期模块名称追加到 lsm_names 字符串中，最后调用 ordered_lsm_init()，完成后返回0，表示初始化成功。

函数ordered_lsm_init()定义为：

static void __init ordered_lsm_init(void)
{
	struct lsm_info **lsm;
	ordered_lsms = kcalloc(LSM_COUNT + 1, sizeof(*ordered_lsms), GFP_KERNEL)
	if (chosen_lsm_order) {
		if (chosen_major_lsm) {
			pr_info("security= is ignored because it is superseded by lsm=\n");
			chosen_major_lsm = NULL;
		}
		ordered_lsm_parse(chosen_lsm_order, "cmdline");
	} else
		ordered_lsm_parse(builtin_lsm_order, "builtin");
	for (lsm = ordered_lsms; *lsm; lsm++)
		prepare_lsm(*lsm);
	init_debug("cred blob size     = %d\n", blob_sizes.lbs_cred);
	init_debug("file blob size     = %d\n", blob_sizes.lbs_file);
	init_debug("inode blob size    = %d\n", blob_sizes.lbs_inode);
	init_debug("ipc blob size      = %d\n", blob_sizes.lbs_ipc);
	init_debug("msg_msg blob size  = %d\n", blob_sizes.lbs_msg_msg);
	init_debug("task blob size     = %d\n", blob_sizes.lbs_task);
	if (blob_sizes.lbs_file)
		lsm_file_cache = kmem_cache_create("lsm_file_cache",  blob_sizes.lbs_file, 0,
					   SLAB_PANIC, NULL);
	if (blob_sizes.lbs_inode)
		lsm_inode_cache = kmem_cache_create("lsm_inode_cache",
					    blob_sizes.lbs_inode, 0,    SLAB_PANIC, NULL);
	lsm_early_cred((struct cred *) current->cred);
	lsm_early_task(current);
	for (lsm = ordered_lsms; *lsm; lsm++)
	initialize_lsm(*lsm);
	kfree(ordered_lsms);
}

这个函数是框架初始化的核心函数之一。它的作用是按照指定顺序初始化所有启用的 LSM，并为它们分配所需的安全扩展（blob）内存。这是最关键的一步，内核会根据配置（如 CONFIG_LSM 宏或内核启动参数 lsm=）确定的优先级顺序，依次调用每个模块的初始化函数。

函数ordered_lsm_init()主要完成确定 LSM 顺序、解析并生成 ordered_lsms 列表、预处理每个 LSM（计算 blob 大小）和分配 slab cache，真正初始化各个 LSM模块。它首先利用kcalloc()为ordered_lsms数组分配内存，通过变量chosen_lsm_order确定LSM 顺序来源，LSM顺序来自内核启动参数或内建默认顺序。

确定LSM顺序来源后，利用ordered_lsm_parse()解析LSM 顺序，把字符串解析成 lsm_info 填入 ordered_lsms数组（每个 lsm_info 表示一个 LSM）。一旦确定顺序，函数通过prepare_lsm()确认各类对象所需最大的安全模块（blob size）。然后通过函数init_debug()打印所有已启用的安全模块所需要的安全数据块（Blob）的累计大小。

在打印完调试信息后，通过函数kmem_cache_create()创建 slab缓存区，为这些对象创建专用 slab。之后利用函数lsm_early_cred()和lsm_early_task() 进行初始化与绑定，确保 0 号进程在安全框架激活前已具备合法的安全上下文空间。最后通过initialize_lsm()真正初始化 LSM，为每个 LSM 注册钩子、设定初始化策略、建立内部结构。
